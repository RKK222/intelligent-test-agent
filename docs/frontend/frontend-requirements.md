# 测试智能体平台前端要求总结

## 1. 产品定位

该平台不是普通测试管理后台，也不是普通聊天机器人，而是一个面向测试领域的 **Web Agent IDE**。

核心目标是让测试人员、测试开发、研发人员可以在一个工作台中完成：

- 查看测试项目目录
- 编辑测试脚本
- 编辑 Python 技能
- 与测试智能体对话
- 查看智能体任务规划
- 查看工具调用过程
- 执行测试
- 查看测试报告
- 分析失败原因
- 查看文件 Diff
- 接受或拒绝智能体修改

一句话概括：

> 这是一个集文件管理、测试执行、报告分析、Agent 对话、技能编辑于一体的测试智能体 Web 工作台。

## 3. 整体页面形态

平台整体应采用 Web IDE 式布局，而不是传统后台式页面。

推荐布局：

```txt
┌──────────────────────────────────────────────────────────────┐
│ 顶部栏：项目 / 环境 / 分支 / Agent / 当前测试运行状态           │
├──────────────┬──────────────────────────────┬────────────────┤
│ 测试目录      │ 主编辑区                      │ Agent 对话区     │
│ 文件树        │ 文件编辑 / Diff / 报告         │ 规划 / 工具卡片   │
├──────────────┴──────────────────────────────┴────────────────┤
│ 底部面板：测试日志 / 失败详情 / 截图 / Trace / 控制台输出        │
└──────────────────────────────────────────────────────────────┘
```

由于测试平台内容较多，需要支持：

* 面板拖拽
* 面板停靠
* Tab 分组
* 浮动窗口
* 弹出窗口
* 保存用户布局
* 恢复上次工作区

因此主工作区建议使用 **Dockview**，而不是简单的分栏拖拽组件。

## 4. 文件与目录能力

平台需要支持测试项目的目录式文件管理。

典型目录包括：

```txt
tests/
fixtures/
reports/
skills/
configs/
playwright.config.ts
pytest.ini
package.json
```

需要关注的前端能力：

| 能力         | 说明                        |
| ---------- | ------------------------- |
| 文件树        | 展示测试项目、技能目录、配置文件、报告文件     |
| 目录展开/收起    | 支持多层级目录浏览                 |
| 文件搜索       | 支持按文件名或内容查找               |
| 多文件 Tab    | 支持同时打开多个文件                |
| 文件查看       | 支持只读查看测试文件、报告、配置          |
| 文件编辑       | 支持编辑测试脚本、配置文件、Python 技能脚本 |
| 未保存状态      | 标记当前文件是否有未保存修改            |
| Agent 修改状态 | 标记哪些文件被智能体修改              |
| Git 状态     | 展示新增、修改、删除等状态             |
| 右键菜单       | 支持新建、重命名、删除、复制路径等操作       |

## 6. Diff 对比能力

Diff 是该平台的核心能力之一，不应作为附属功能处理。

智能体可能会修改：

* 测试脚本
* Python 技能脚本
* 测试配置
* 测试数据
* Agent Prompt
* 工具参数

用户需要清楚看到：

* 改了哪个文件
* 新增了什么
* 删除了什么
* 为什么改
* 是否接受
* 是否拒绝
* 是否让 Agent 重写

建议使用 **Monaco Diff Editor**。

Diff 入口建议包括：

| 入口               | 说明                            |
| ---------------- | ----------------------------- |
| 文件树修改标识          | 点击被修改文件直接查看 Diff              |
| Agent 对话卡片       | 在 DiffActionCard 中点击“查看 Diff” |
| Changed Files 面板 | 汇总展示所有变更文件                    |
| 测试失败分析卡片         | 从失败分析跳转到建议修改 Diff             |
| 技能编辑器            | 查看技能脚本版本差异                    |

Diff 操作需要支持：

* 查看 Diff
* 接受当前文件修改
* 拒绝当前文件修改
* 接受全部修改
* 拒绝全部修改
* 复制 Diff
* 回到源文件
* 让 Agent 重新生成修改

## 7. Agent 对话能力

右侧 Agent 对话区不是普通聊天窗口，而是测试智能体操作台。

需要支持：

| 能力         | 说明                     |
| ---------- | ---------------------- |
| 用户输入测试任务   | 用户可以自然语言描述测试目标         |
| Agent 流式回复 | 智能体逐步输出分析和执行过程         |
| 任务规划展示     | 用结构化卡片展示 Agent 的执行计划   |
| 工具调用展示     | 展示读文件、执行测试、分析报告等工具调用   |
| 可操作卡片      | 对话中可直接点击按钮操作           |
| 技能调用展示     | 展示 Python 技能调用过程       |
| 人工确认       | 对文件修改、测试执行、技能发布等动作进行确认 |
| 多轮上下文      | 支持围绕报告、失败用例、Diff 继续追问  |

建议使用 **assistant-ui** 来承载 Agent 对话、工具调用和可操作卡片。

## 8. 对话中的卡片类型

Agent 对话中应支持结构化卡片，而不是只展示纯文本。

建议卡片类型如下：

| 卡片                  | 作用                          |
| ------------------- | --------------------------- |
| PlanCard            | 展示 Agent 的测试任务规划            |
| ToolCallCard        | 展示工具调用，例如读文件、查报告、跑测试        |
| TestRunCard         | 展示测试执行状态和进度                 |
| ReportSummaryCard   | 展示测试报告摘要                    |
| FailureAnalysisCard | 展示失败原因分析                    |
| DiffActionCard      | 展示 Agent 建议修改，并提供查看/接受/拒绝按钮 |
| SkillCallCard       | 展示 Python 技能调用过程            |
| SkillErrorCard      | 展示技能执行错误和修复入口               |
| ApprovalCard        | 展示需要人工确认的操作                 |
| DependencyCard      | 展示技能依赖缺失或安装状态               |
| TraceCard           | 展示失败用例 Trace 相关入口           |

卡片中的操作按钮可以包括：

* 打开文件
* 查看日志
* 查看报告
* 查看截图
* 查看 Trace
* 运行测试
* 停止测试
* 重跑失败用例
* 查看 Diff
* 接受修改
* 拒绝修改
* 打开技能
* 重新运行技能
* 创建缺陷
* 让 Agent 重新分析

## 9. 测试执行能力

平台需要支持从用户或 Agent 触发测试执行。

需要支持：

| 能力     | 说明                        |
| ------ | ------------------------- |
| 选择测试范围 | 全量、模块、文件、失败用例、Agent 推荐范围  |
| 选择执行环境 | dev、staging、preprod、本地容器等 |
| 启动测试   | 手动启动或 Agent 调用工具启动        |
| 停止测试   | 支持取消当前测试任务                |
| 实时进度   | 展示当前执行数量、通过数、失败数、跳过数      |
| 实时日志   | 展示测试执行日志                  |
| 执行结果   | 展示最终测试结果                  |
| 重跑失败   | 支持重跑失败用例                  |
| 执行历史   | 保存每次测试执行记录                |

测试执行状态建议标准化：

```txt
queued
running
passed
failed
skipped
cancelled
timeout
flaky
```

## 13. Agent Event Stream 设计

前端不应该只接收 Agent 的文本，而应该围绕结构化事件流设计。

建议支持事件：

```txt
message.delta
plan.updated
tool.started
tool.output
tool.completed
tool.failed
test.started
test.progress
test.completed
report.generated
file.changed
file.diff
skill.started
skill.output
skill.completed
skill.failed
approval.required
run.completed
```

事件驱动的 UI 行为：

| 事件                | 前端行为                           |
| ----------------- | ------------------------------ |
| message.delta     | Agent 对话流式追加文字                 |
| plan.updated      | 更新 PlanCard                    |
| tool.started      | 新增 ToolCallCard                |
| tool.output       | 更新工具调用输出                       |
| tool.completed    | 标记工具调用成功                       |
| tool.failed       | 标记工具调用失败                       |
| test.started      | 新增 TestRunCard                 |
| test.progress     | 更新测试执行进度                       |
| test.completed    | 生成测试报告入口                       |
| report.generated  | 打开 ReportPanel                 |
| file.diff         | 打开 DiffPanel 或更新 Changed Files |
| skill.started     | 新增 SkillCallCard               |
| skill.completed   | 展示技能输出                         |
| skill.failed      | 展示 SkillErrorCard              |
| approval.required | 展示 ApprovalCard                |
| run.completed     | 标记当前 Agent Run 完成              |

## 14. 状态管理要求

建议分清两类状态：

### 前端界面状态

由 **Zustand** 管理。

包括：

* 当前打开的文件
* 当前选中的测试 run
* 当前选中的失败用例
* 当前选中的技能
* 当前 Diff 文件
* 当前展开的工具卡片
* 当前 Dockview 布局
* 当前 Agent 面板状态
* 当前底部日志面板状态

### 后端服务数据

由 **TanStack Query** 管理。

包括：

* 文件树
* 文件内容
* 测试执行记录
* 测试报告
* 失败用例
* 技能列表
* 技能详情
* 技能运行结果
* Agent 对话历史
* 工具调用历史

## 16. 前端最需要关注的要点

| 关注点        | 说明                                                                          |
| ---------- | --------------------------------------------------------------------------- |
| Web IDE 布局 | 使用 Dockview 支持面板拖拽、停靠、悬浮、弹出、布局保存                                            |
| 文件管理       | 支持测试文件、配置文件、报告文件、技能文件的目录式查看和编辑                                              |
| 代码编辑       | 使用 Monaco Editor 编辑测试脚本、Python 技能、JSON/YAML 配置                              |
| Diff 对比    | 使用 Monaco Diff Editor 展示 Agent 修改，并支持接受/拒绝                                  |
| Agent 对话   | 使用 assistant-ui 展示流式对话、任务规划、工具调用和可操作卡片                                      |
| 测试执行       | 展示执行状态、进度、日志、失败信息、重跑入口                                                      |
| 测试报告       | 展示摘要、趋势图、失败表格、截图、Trace、Agent 分析                                             |
| 技能编辑       | 支持 Python 技能脚本编辑、调试运行、依赖配置、版本管理                                             |
| 卡片交互       | 对话中支持 PlanCard、ToolCallCard、TestRunCard、FailureAnalysisCard、SkillCallCard 等 |
| 状态管理       | Zustand 管理界面状态，TanStack Query 管理后端数据                                        |
| 事件驱动       | 前端围绕 Agent Event Stream 更新 UI，而不是只展示文本                                      |
| 布局恢复       | 用户自定义工作区布局需要可保存、可恢复、可重置                                                     |
| 信息密度       | 内容很多，需要通过 Dock、Tabs、浮窗、Drawer 降低单页拥挤度                                       |

## 17. 一句话总结

这个前端最重要的不是“页面多”，而是要把：

```txt
文件
编辑器
Diff
Agent 对话
任务规划
工具调用
测试执行
测试报告
失败分析
Python 技能编辑
技能调试
日志与 Trace
```

放进一个可拖拽、可悬浮、可操作、可恢复布局的 **测试智能体 Web IDE 工作台** 中。
