# 工作区 Git 面板交互修复设计

## 目标

修复工作区 Git 面板在真实 merge 冲突下的暂存、文件预览和冲突保存问题，同时保留 Git 原生约束：只要仍有 unmerged 文件，就不能提交或推送。

## 设计

### 真实暂存与取消暂存

新增工作区 `git-stage` 与 `git-unstage` API。前端不再只修改本地 `Set`，而是通过后端把工作区相对路径映射为仓库相对路径后执行定点 `git add` 或 `git restore --staged`，完成后重新读取 `git-diff`。

冲突期间允许对非冲突文件执行 stage/unstage；冲突文件只能通过冲突解决 API 写入并 stage，普通暂存 API拒绝 unmerged 路径。只要 `git-diff` 仍返回冲突文件，提交和提交并推送按钮保持禁用，并通过按钮提示说明 Git 不允许在未解决冲突时提交。

### 文件名与 Diff 打开性能

所有 staged、unstaged、conflict 和 Agent 文件行把完整路径挂在整行 `title` 与可访问名称上，鼠标停留在行内任意位置都能看到完整路径。

点击工作区文件时，`GitChangesPanel` 把当前已加载的文件 Diff 一并交给工作台。工作台直接切换到 VCS Diff 并选中该文件，不再先请求整份 `git-diff`；只有缓存中找不到目标文件时才回退刷新。这样避免先出现空的 VCS 来源状态和重复网络/Git 扫描。

### 冲突结果保存

`MergeConflictEditor` 使用 Vue `ref` 持有合并结果，而不是让计算属性直接读取非响应式 Monaco model。Monaco 内容变化、保留当前、采用应用、保留两者和删除都会同步更新该结果。

“保存并标记已解决”仅在结果已修改且不含 Git 冲突标记时启用；点击后发送当前结果。初始化尚未完成时点击预设按钮也会先更新响应式结果，Monaco 完成加载后使用该结果，避免点击丢失。

## 错误处理与兼容性

- 新 API 复用现有工作区归属、路径规范化、SSH key 和统一错误响应。
- 请求体继续复用 `{ "files": [...] }`，不新增数据库、事件或 generated SDK 变更。
- stage/unstage 任一文件为冲突路径时整次请求失败，不做部分执行。
- 前端 API 失败后保留原列表并展示错误，成功后以服务端 `git-diff` 为准。

## 测试

- 后端业务层和 Controller 测试覆盖真实 stage/unstage 调用、冲突路径拒绝和路由。
- 前端组件测试覆盖冲突期间可 stage/unstage、提交仍禁用、整行完整路径提示和 API 刷新。
- 工作台测试覆盖缓存命中时不重复读取 `git-diff`。
- 合并编辑器测试覆盖预设选择后保存按钮启用并提交正确内容。
- 使用现有本地真实冲突 worktree 做运行态验证。
