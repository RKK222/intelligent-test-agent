# 数据库 IDENTITY 运维功能设计

## 背景与目标

添加用户时报错"数据冲突：当前操作因存在关联数据无法执行，请先清理关联记录后重试"。根因是 `users` 表的 `id` 列使用 PostgreSQL `GENERATED BY DEFAULT AS IDENTITY`，数据库中已存在 id=1..N 的用户行（测试造数据/导入），但 identity 序列未被同步推进，仍从低位生成。`JdbcUserRepository.save` 的 insert 不显式指定 id，依赖 `nextval`，于是生成已存在的 id（如 8），命中 `users_pkey` 唯一约束，抛 `DuplicateKeyException`（`DataIntegrityViolationException` 子类），被 `GlobalExceptionHandler.handleDataIntegrityViolation` 统一翻译成上述误导性文案。

历史上 `user_roles` 表出现过完全相同的序列错位问题，由 migration `V20260627214000__reset_user_roles_identity_sequence.sql` 用 `restart with 1000000` 修复。

本功能把"查询 identity 当前值 + 手工滚动 identity"做成超管可自助操作的运维入口，覆盖当前 `users` 表问题及未来同类问题，避免每次都靠加 Flyway migration 临时修复。

## 作用范围

固定白名单多表。后端预置 `IdentityManagedTable` 枚举，初始纳入 4 张身份授权核心表：

- `USERS` → `users`
- `USER_ROLES` → `user_roles`
- `DICTIONARIES` → `dictionaries`
- `USER_LOGIN_LOGS` → `user_login_logs`

枚举持有真实表名。identity 序列名不硬编码，运行时由 `pg_get_serial_sequence(#{tableName}, 'id')` 动态取得（PG `GENERATED ... AS IDENTITY` 默认为 `<table>_id_seq`，但动态取避免序列名非默认时出错）。Service 只接受枚举，杜绝任意表名注入。后续按需扩展白名单即可，扩展成本极低。

## 架构与模块归属

严格贴合 AGENTS.md 分层与第 24 条（新增 SQL 必须走 MyBatis XML mapper）：

```text
test-agent-api (Controller)
  └─ DatabaseIdentityController        // HTTP 入口，鉴权 + 参数转换，仅 SUPER_ADMIN
       ↓
test-agent-system-management (应用服务)

       ↓

  └─ DatabaseIdentityMapper (interface) + DatabaseIdentityMapper.xml

```

- Controller 放 `test-agent-api` 的 `web/platform` 包，与 `UserManagementController` 同包，复用 `requireSuperAdmin`。
- 应用服务放 `test-agent-system-management` 的 `maintenance` 子包。
- Mapper 接口与 XML 放 `test-agent-persistence/.../mybatis`，遵循现有 `CommonParameterMapper` 范式。
- 依赖方向：api → system-management → persistence（domain），符合 `dependency-rules.md`。

## API 设计

3 个 HTTP 接口，挂在 `/api/internal/platform/system-management/identity`，仅 SUPER_ADMIN：

| 方法 | 路径 | 作用 | 请求体 | 响应 |
|---|---|---|---|---|
| GET | `/identity` | 查询白名单所有表 identity 状态 | 无 | `List<IdentityStatusDto>` |
| POST | `/identity/align` | 指定表 identity 对齐到 max(id)+1 | `{table: "USERS"}` | `IdentityStatusDto` |
| POST | `/identity/restart` | 手动指定目标值 RESTART WITH n | `{table: "USERS", targetValue: 1000000}` | `IdentityStatusDto` |

`IdentityStatusDto` 字段：

- `table`：表显示名（枚举名，如 `USERS`）
- `tableName`：真实表名（如 `users`，仅展示用）
- `currentValue`：identity 序列当前 `last_value`（表空时为 null）
- `maxId`：表中当前 `max(id)`（表空时为 null）
- `conflict`：布尔，`currentValue != null && currentValue < maxId` 表示错位
- `lastUpdatedAt`：本次查询时间

## 数据流

**查询（GET /identity）**：Controller 鉴权 → Service `listIdentityStatuses()` → 遍历白名单枚举调 `mapper.queryIdentityStatus` → 每张表一条 SQL：先用 `pg_get_serial_sequence(#{tableName}, 'id')` 取序列名，再从 `pg_sequences` 按 `sequencename = #{sequenceName}` 取 `last_value`，同时 `select max(id) from ${tableName}` 取最大主键。

**对齐（POST /identity/align）**：Controller 鉴权 + 校验 `table` 在白名单 → Service `alignIdentity(table)` → 先查旧值记审计日志 → 取 `max(id)` → `ALTER TABLE ${tableName} ALTER COLUMN id RESTART WITH #{maxPlusOne}` → 返回刷新后状态。

**手动重置（POST /identity/restart）**：同 align，但 `targetValue` 来自请求体，Service 校验 `targetValue > maxId`（不允许往回滚造成新冲突）后 `RESTART WITH ${targetValue}`。

### SQL 注入与安全约束

`RESTART WITH` 后只能接字面量，不能用 JDBC 参数绑定，也不能接子查询。因此：

- **表名** `${tableName}` 必须由 Service 传入白名单枚举的真实表名常量，非用户输入。
- **目标值** `${targetValue}` 由 Service 层 `Long` 校验（正整数且 `> maxId`）后拼接，杜绝越界/回退。
- `current_value` / `sequenceName` 用 `#{}` 参数绑定。

**审计**：每个写操作前后用 `LOGGER.info` 记录 `event=identity_maintain traceId=xxx operatorUserId=xxx table=xxx action=ALIGN|RESTART oldValue=xxx maxValue=xxx newValue=xxx`。不新建审计表。

**错误处理**：表名不在白名单 → `PlatformException(VALIDATION_ERROR, "不支持的数据表")`；targetValue 非正整数或 `<= maxId` → `VALIDATION_ERROR`；复用现有 `ErrorCode`，不新增枚举。

## 前端设计

入口：现有 `SettingsUserManagementPanel.vue` 内底部新增折叠区块「数据库 IDENTITY 运维」，仅 `hasPermission`（SUPER_ADMIN）时渲染。

交互：

- 表格展示 `IdentityStatusDto[]`：表名 / 序列当前值 / 最大 ID / 是否错位 / 操作。
- `conflict=true` 的行整行高亮，提示"序列落后于已有主键，新增可能冲突"。
- [对齐]：`ElMessageBox.confirm` 二次确认 → `alignIdentity(table)` → 成功刷新该行。
- [重置]：`ElMessageBox.prompt` 输入目标值，前端校验正整数且 > maxId → `restartIdentity(table, targetValue)` → 成功刷新。
- 错误复用面板现有 `errorMessage` 模式 + `ElMessage.error`。

`backend-api` 封装（紧接现有 `listUsers/createUser/listRoles`）：

```ts
listIdentityStatuses: () => request<IdentityStatus[]>(`${systemManagementBase}/identity`),
alignIdentity: (table: string) => request<IdentityStatus>(`${systemManagementBase}/identity/align`, { method: "POST", body: { table } }),
restartIdentity: (table: string, targetValue: number) => request<IdentityStatus>(`${systemManagementBase}/identity/restart`, { method: "POST", body: { table, targetValue } }),
```

`shared-types` 新增 `IdentityStatus` 类型，对应后端 DTO 字段。

## 兼容性与安全

- 不修改表结构，无需 Flyway migration。
- 复用现有 `ErrorCode`，不新增枚举。
- 不新增依赖。
- 不修改 `.env.local` 等环境配置文件。
- 表名走白名单常量、目标值走 Long 校验，杜绝 SQL 注入与回退。
- 仅 SUPER_ADMIN 可访问，双保险（菜单层 + Controller 鉴权）。
