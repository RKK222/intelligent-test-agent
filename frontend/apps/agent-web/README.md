# @test-agent/agent-web

## 工程定位

首次登录引导按普通用户的真实路径说明应用下拉、workspace/version 选择、小地球引入需求子条目和首条消息自动建对话；引导完成后可从用户手册重新播放。

Vue 3 + Vite SPA 主应用，组合 Web IDE 工作台、文件树、Monaco 编辑器、Agent 对话、Run 面板和 Diff 查看。

## 主要职责

- 夜间异步执行交互由 `AgentWorkbench` 与 `FigmaChatPanel` 组合：输入框发送按钮左侧提供定时图标，点击后加载北京时间 21:00 至次日 07:00 的 15 分钟容量时段并默认选中后端推荐值；新对话草稿成功提交后才清空输入。主对话始终展示当前会话内容；“会话列表”以 Teleport 非模态抽屉覆盖对话栏左侧编辑区，内部提供“会话 / 待执行任务”页签，选择会话或任务后保持打开，当前会话高亮，再次点击“会话列表”、关闭按钮、Esc 或收起右栏时关闭。待执行页逐页收齐全部待执行项，并展示内容、时段、创建时间和顺延状态。当前 Session 存在 `SCHEDULED/DISPATCHING` 任务时禁用普通发送但保留“新建对话”。切换到待执行页签时立即查询，页面另按每 30 秒及窗口重新聚焦刷新；容量冲突会重取最新时段。成功投递后复用现有 Session/Run/RunEvent 展示，并以来源标签和北京时间实际启动时间区分。

- 提供 `/login` 登录页、`/` 工作台页面和 `/s/[sessionId]` 只读 transcript 页面（vue-router 客户端路由），未知路径回退到工作台。
- 组合 dockview-vue 三栏布局和底部运行面板。
- 组合 Figma Web IDE 风格的 40px 顶栏、48px activity rail、紧凑左侧文件面板、中间编辑器和右侧 Agent 面板；Run/Terminal 默认隐藏在底部抽屉中，通过 activity rail 打开。
- 左侧活动栏底部、系统设置按钮上方的小宠物开关与当前值班伙伴使用同一组头像素材。伙伴目录包含寻迹鹿、巡查红熊猫、深夜龙、星探狐、稳态熊猫、工具浣熊和夜巡猫七种角色；浮层标题栏的伙伴按钮可选择“每日轮换”“每天随机”或“自主选择”，并可用宠物大小滑杆调整 75%–250% 的显示比例（默认 150%）。每日轮换按本地自然日顺序切换，每天随机在当天保持稳定并尽量避免连续两天重复，自主选择固定用户指定角色；伙伴、大小偏好与宠物位置、固定状态一样只保存在浏览器本地，不新增用户配置 API。宠物可立即唤起或收起，连续 60 秒无鼠标、键盘、滚动或点击且页面聚焦可见时也会自动出现。首次唤起和首次自动出现默认固定，避免随机跳跃；用户可双击宠物取消固定，取消后才恢复自然随机动作与自然离场。手动唤起后保持可见，直到用户收起或重新定位。保存坐标只作为下一次出现的起点，并额外支持快速抖动和旋转庆祝。点击进程 ready 的宠物打开以对话为主体的统一浮层：对话仍按真实主 Session 控制旁路提问，标题栏低强调的小手柄按钮可进入不依赖 Session 的游戏区；四个紧凑 2×2 小卡片分别提供 10×16 俄罗斯方块、8×8/10 雷扫雷、9×9 数独和 12×12 贪吃蛇，不再增加独立活动栏游戏按钮。扫雷支持在相邻旗子数与数字匹配时双击数字格，自动展开其余相邻区域；旗子数不匹配时保持棋盘不变。统一浮层打开期间宠物不会自然离场，关闭、收起宠物或切换游戏时释放或暂停对应游戏计时器。七种角色统一用头像外沿的细光圈承载进程状态：可用为青蓝色，未初始化或不可用为暖红色，检查中为银蓝色；光圈只描边不填充，名册场景按角色使用不同底色，状态不再叠加到眼睛或独立绿点。已分配的 opencode 进程终止后，点击活动栏宠物入口会直接复用现有初始化流程启动进程；启动完成后只唤出浮动宠物，不再要求用户先唤出宠物再点击启动。首次未分配或其它未 ready 状态仍通过进程状态气泡说明并初始化。聊天区不再显示独立的进程绿点/状态卡，子 Agent 视图也不显示进程入口；这些交互不新增或变更 API、RunEvent SSE、数据库、安全或兼容性契约。
- 小宠物首次点击打开的对话页提供“Agent 配置更新”入口；进入小宠物选择页时只展示伙伴、轮换方式和大小设置，不展示运行态更新按钮。左侧 Agent 区域的公共级/应用级根节点仍保留同一组按钮，刷新图标统一放在各自动作组的最右侧。按钮点击后先说明将切换/释放哪一类个人运行态，确认后才执行；公共级调用既有公共个人 runtime-reload 接口并携带当前用户的 worktree/server，应用级只调用既有 global dispose。公共入口仅超级管理员可见，应用入口仅应用管理员或超级管理员可见；按钮按用户级 `runningCount` 和统一重载锁禁用，后端若在竞态窗口返回 `CONFLICT`，自动保存会保留待处理代次并在用户空闲后重试，不影响其他用户，不新增接口。
- 工作台首页“开始您的探索”区域和顶部问号都可打开内置 VitePress 用户手册，手册和本地全文搜索始终可用；应用内 Help 左侧固定章节清单与 VitePress 导航同步包含“开发与测试目录”和“设置与权限内操作”，这些章节的首页入口、内嵌页面和宠物问答共同读取同一 Markdown。首次登录首屏在视口中央介绍整个工作面板，随后依次框选应用、workspace/version、需求子条目、对话、宠物、设置和手册七个普通用户主入口；设置流程会按权限拆成 SSH 配置、应用与版本库配置、应用工作区配置三个独立步骤，进入每步时自动打开设置弹窗并锚定真实菜单项，普通用户不展示应用管理员步骤，也不展开超级管理员专属流程。完成后打开手册，手册内可重新播放。完成状态按“浏览器本地存储 + 用户 ID + 引导版本”记录，因此换浏览器、清理站点数据或升级引导版本后会再次展示。帮助中心右侧的小宠物问答会把当前 Markdown 章节作为限定资料：有主 Session 时复用临时 fork，无主 Session 时按当前工作区启动独立手册问答 Run，不创建普通对话。
- 通过 `@tanstack/vue-query` 调用 `@test-agent/backend-api`。
- 首次当前用户进程查询保持无路由提示；拿到 `linuxServerId` 后，工作台、只读 transcript、设置中的本地工作区、Git/Agent 配置子面板和两个 fetch SSE 共用页面内存中的动态路由 ID。退出、认证 token 变化或刷新即清空，不写 localStorage/sessionStorage；共享控制面请求不使用该值。
- 新用户新手引导展示期间，宠物只保留引导锚点，不自动弹出 TestAgent 进程状态面板；引导完成或关闭后再恢复自动提示，用户主动打开宠物入口不受影响。
- 小宠物的进程状态气泡可继续打开旁路问答浮层；工作台调用 `startSideQuestionRun` 后复用 RunEvent SSE 展示真实上下文准备、只读检查和答案增量。等待时点击工作区不会关闭，切换会话、收起宠物、显式关闭或卸载只释放旁路订阅，不中止主 Run。已登录页面的 RunEvent SSE 使用带 `Authorization` 的 fetch 流式读取（token 只在请求头，不进入 URL），以便主对话、历史恢复后的继续订阅和宠物旁路都能通过后端跨节点路由鉴权；断流仍按最后一个 SSE `id` 自动续传。后端在临时 fork 中按需 compact，固定使用 `build` agent 并以系统提示约束为只读，最终答案仅显示在浮层，不进入主对话时间线。
- 小宠物拖动和缩放按企业内 Chromium 108 兼容：拖动期间由 `window` 捕获阶段统一接收 Pointer Events，即使编辑器或工作台子组件停止事件冒泡也能继续移动和保存位置，不调用 pointer capture；大小使用普通 range input 和根元素 width/height 计算，不使用新版 CSS `scale` 属性；键盘方向键移动保持不变。
- 宠物旁路问答有两种模式：已有主 Session 时追问当前上下文；无主 Session 时用内置手册默认章节约束问题，并调用 `startManualQuestionRun`。两种模式都要求 TestAgent 进程可用；手册模式还要求当前工作区已选中，页面会明确显示“用户手册 · 不创建主对话”。
- 当进程 ready 且主 Session 或工作区可用时，单击宠物直接打开并聚焦提问输入框；进程未就绪时仍展示状态卡及初始化提示。
- 设置页按角色开放真实能力：`SUPER_ADMIN` 可新建应用并管理用户；`APP_ADMIN` 可见应用管理和版本库管理，维护已有应用成员、关联与工作空间；普通用户只使用个人设置配置自己的 SSH Key。新建应用使用配置管理 API，工作空间仍复用既有异步创建进度。
- 运行管理中的 worker 容器以 `containerName` 显示可读 hostname，拓扑图通过 `containerId` 关联容器与 manager；`containerId` 只作为不透明路由键保留在 tooltip、筛选、指标和控制命令中，旧响应缺少 `containerName` 时才回退显示 `containerId`。服务器/Java 与容器/manager 合并表紧邻对应进程 ID 展示 `buildVersion`，滚动升级期间每行展示自身构建版本，旧进程或旧快照缺失时显示 `-`。
- 应用和个人工作区切换只使用后端返回的运行态 Workspace id；普通前端不传物理目录创建 Workspace，切换时清空旧文件树、编辑器、Diff、Session/Run、队列和聊天运行态。
- 通过 `@test-agent/event-stream-client` 订阅 RunEvent SSE；订阅身份只由标量 `(runId, sessionId, token)` 决定，同一逻辑 Run 的 status、标题待定标记或其它 Run 对象投影变化不得重建连接。收到终态时先建立 500ms hold 再更新 `run.status`，使旧后端紧邻重放的 `run.failed → run.succeeded` 能在原连接内完成纠正；普通终态在 hold 结束后关闭，`platformSessionTitlePending=true` 时继续复用原连接直到标题同步或 watch closed。事件只作用于当前活动 Run，新 Run、Session 或 token 变化各自只切换一次，旧订阅晚到的 `run.cancelled/run.succeeded/run.failed` 不会覆盖下一轮运行态。legacy 终态反馈恢复按 `runId` 合并，同一 Run 最多启动一条 `messages → run-feedbacks/me/query` 链，链内仍保持 500ms 间隔、最多 3 轮。`session.status.retry` 会保持当前 Run 为运行中，在时间线展示错误内容和 60 秒倒计时；第 1/2 次倒计时到期后前端会用最近一次 Run 草稿自动重试，第 3 次倒计时结束仍未恢复时本地标记对话失败。
- 订阅既有 wire name `session.updated`：只消费后端在平台标题成功持久化后追加的 `platformSessionTitleSynchronized: true` 与 `platformSessionTitle`，不再从 `info` 或 `rawPayload` 推断标题。仅当事件所属订阅 Run 的平台 Session 仍为当前会话、事件不是 child session 时，更新当前会话和已加载历史会话的标题，并失效 sessions cache；切换历史会话后到达的旧订阅事件，以及没有平台确认标记的 child、空标题、不匹配或持久化失败事件，都不覆盖当前标题。该消费不新增 API、DTO、事件 wire name、数据库/Flyway 或 SDK，旧客户端可忽略该既有事件。
- 收到 transient `run.snapshot.reset` 时，agent-chat reducer 原子完成当前 Run 运行投影清空与 snapshot 重放；`AgentWorkbench` 同步清空独立维护的 Run Diff、实时跟随工具 part 和已选 Diff 路径，再按 snapshot events 原顺序重建页面副作用且不重复桌面通知。reset/snapshot events 不更新 durable Last-Event-ID，空或旧字段缺失按空快照兼容。
- 通过 `@test-agent/event-stream-client` 的用户级运行态 fetch SSE 订阅历史会话运行中计数和 `question.asked` 待回答提醒，并把摘要中的 busy Run 作为恢复 RunEvent SSE 的主入口；连接失败时每个故障窗口、每个 Session 只允许一次 `active-run` fallback，不做固定间隔热轮询。运行中点击“新建对话”只清空当前视图并关闭当前 RunEvent SSE，不调用 cancel/abort，后端继续接收 opencode 事件，历史切回时优先使用 runtime-state 摘要与 session-tree/messages 恢复。
- 对话回放可重复 fixture 位于 `tests/workbench.spec.ts`：`installPetSideQuestionRunEventStream` 以带认证 fetch SSE mock 覆盖宠物进度、断流续传、重复增量和失败重试；`installAuthenticatedRunEventFetchStream` 覆盖冲突终态重放、标题等待、单连接约束、新 Run 切换和 legacy 反馈恢复上限；同文件还覆盖空白对话创建夜间任务、刷新后从会话列表进入待执行页签、模拟后台投递后抽屉保持打开并回到既有消息/Run 展示，以及连续切换会话时抽屉保持可见。历史 native question 场景覆盖从会话列表进入旧会话后恢复回答 dock。部件级 Question、Permission、Todo 与子 Agent fixture 分别位于 `tests/FigmaChatPanel.test.ts`、`packages/agent-chat/tests/` 和 `tests/opencode-parts-real-e2e.test.ts`，不得把这些测试数据写入 Flyway 或生产库。
- 在 `AgentWorkbench` 内维护会话级原始报文内存缓存，每个 Session 仅保留最新 2000 条；右侧 `FigmaChatPanel` 的“原始输出”浮层展示当前页面生命周期内捕获的会话相关 HTTP 请求/响应正文和 RunEvent fetch SSE `data`，并支持按请求/响应/SSE 类型和本地关键词搜索过滤标题、URL、traceId、runId 与正文，命中的可见文本会高亮；浮层提供下载按钮，将当前过滤结果导出为本地文本文件，无可见报文时按钮禁用。该能力仅用于前端调试，不新增后端 API、表结构或 RunEvent 契约；导出内容沿用进入页面缓存前已递归脱敏 `contextToken` 并截断后的可见副本，不绕过安全边界。fetch SSE 只保存客户端实际解析到的 `data` 原文。若 transport 报告连接异常但后端尚未发出 `run.failed/run.succeeded/run.cancelled`，前端会在原始输出中追加一条本地 SSE 诊断记录，并在时间线展示一次连接异常提示，等待客户端自动重连。
- 用 Pinia 工作台状态管理打开文件、活动 tab 和 Diff 选择。
- 承载"实时追踪"行为：用户开启后，运行中的 `message.part.updated`/`diff.proposed` 会打开最近变化文件、刷新只读预览，并把 `files[].additions/deletions` 传给文件树展示行数。所有 `diff.proposed` 都会立即强制刷新变更文件的父目录和工作区 Git Diff，即使 Run 尚未结束或实时追踪关闭，新建/删除文件也无需刷新整页即可出现在文件树和 Git Changes。`workbench-utils.inferDiffFromToolPart` 在 opencode 1.17.8 `session.diff` 事件 diff 字段为空时，基于 `write`/`edit`/`apply_patch`/`str_replace`/`multi_edit`/`create_file` 工具的 input 估算新增/删除行数并补齐 `RunDiffFile`，让"文件变更卡片 +N"在写盘后即时刷新；该推断与"实时追踪 toggle"解耦，关闭实时追踪时仍会同步 diffFiles 和文件树，但不再打开只读预览 tab。

## 禁止事项
