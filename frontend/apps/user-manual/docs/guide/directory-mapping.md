# 开发与测试目录映射

本页把研发侧 SDD/智能研发目录，映射为测试工作台中的页面目录和实际 Git worktree。它同时说明当前实现与目标设计：当前应用 Agent 配置和业务文件仍在同一个个人 worktree；目标结构会把应用 AI Git、业务开发 Git 和本地 `spec` 工作区分开治理。

::: warning 设计状态
“当前”来自平台现有实现；“目标”是依据智能研发目录、应用 AI Git 和本地 `spec` 约束整理的目录设计，尚不代表相关仓库拆分及组装层已经上线。
:::

## 三条目录边界

### `docs` 是应用级稳定测试资产

`docs/业务`、`docs/功能`、`docs/架构` 由应用 AI Git 单独管理。它们不是业务开发 Git 的完整副本，而是经过来源确认、结构化整理和应用管理员审核后，可以跨需求复用的测试事实。

- `业务`：解释为什么这样做，包括领域术语、角色、流程、业务规则和产品口径。
- `功能`：解释系统做什么，包括需求基线、功能清单、接口、页面、数据和异常口径。
- `架构`：解释系统怎样实现和运行，包括模块、依赖、数据流、部署、安全及非功能约束。

每份稳定资产应记录来源、适用版本、负责人和更新时间。Agent 默认只装载轻量目录清单，正文在任务需要时按路径读取，避免把全部材料塞入每轮上下文。

### `spec` 是当前工作的本地事实

`spec/<需求项>/` 聚合当前需求在 `01-需求`、`02-设计`、`03-编码`、`04-测试` 四个阶段的内容。它允许当前用户和 Agent 在本地 worktree 提交形成检查点，但无论用户角色都不设置远端跟踪分支，也不允许推送。

代码阶段优先保存业务开发 worktree 的路径、提交号、Diff 或必要的设计说明，不复制整个源码仓库。测试阶段保存案例、脚本、执行证据和结论。

### `.opencode` 与 `rules` 只定义行为

应用 AI Git 使用 `.opencode/agents/` 和 `.opencode/skills/` 让 OpenCode 原生发现应用 Agent 与 Skill；`AGENTS.md` 和 `rules/` 承载应用级约束。模型、Provider、通用 MCP 和全局安全配置继续由公共 `opencode.jsonc` 统一提供，应用仓库默认不重复维护 `opencode.jsonc`。

## 发布与更新规则

1. 公共 Agent/Skill 由平台团队建设，超级管理员发布。
2. 应用 Agent、Skill、Rules 和 `docs` 由应用团队建设，应用管理员及以上角色提交并推送。
3. 应用 AI Git 发布后，其他用户的运行态默认切换到新的发布提交；有本地编辑的 worktree 不被静默覆盖。
4. `spec` 只允许当前个人 worktree 本地提交，不进入任何远端发布流程。
5. 所有编辑、发布和任务执行都在 worktree 中进行；运行态目录也使用只读发布 worktree，而不是直接操作裸仓库或共享主工作树。
