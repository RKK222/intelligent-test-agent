# test-agent-opencode-runtime

## 工程定位

与 agent 运行相关的后端业务编排模块，承载 Session、Run、RunEvent 编排、通过 `AgentRuntimeRegistry` 调用 agent、Diff/revert 和受控 PTY terminal 业务；当前唯一真实 agent 实现为 opencode。

## 主要职责

- Agent 配置 rollout 以 `config_scope=PUBLIC/APPLICATION` 区分发布流程，而不是新增 OpenCode 配置覆盖层：公共范围单独互斥，应用范围按版本 ID 互斥，不同应用发布不会再占用公共发布锁。公共范围由公共配置服务同步 Git 运行副本并登记本机全部存量进程；应用范围由托管工作区服务先把指定 feature 提交投影到干净的个人 worktree，再只登记同步成功用户的进程。脏工作区或合并冲突持久化为独立 `AWAITING_USER` 补偿任务，主 rollout 完成后仍会按 worktree 租约继续尝试；收敛后仅为该用户登记 dispose 目标。两者复用同一进程身份核验、Session 空闲检查、用户消息闸门和 OpenCode 原生 `/global/dispose`。

- `PublicAgentConfigRolloutService` 实现公共 Agent/Skill 发布的 `PREPARING -> DRAINING -> COMPLETED/ABORTED` 持久化消息闸门与持续排空 worker。发布成员使用独立 membership 表：新版 Java 自动登记，临时离线成员继续参与，`linux_servers` 历史行不会误阻塞；超级管理员只能显式退役已经停止 Java 和 manager 的成员。每台服务器先用带 fencing token 的数据库租约认领 Git 同步，再取得本服务器 manager 快照；进程清单必须包含 PID、启动时间和 baseUrl，用户归属只查询本 `linuxServerId` 在分页进程仓储允许的 200 条上限，按服务器/容器/端口/PID/启动时间匹配，其他服务器历史脏行不能阻塞本机登记，无法归属的本机目标会保持全员门禁直到安全 dispose。所有服务器确认后，每个 Java 只认领本 `linuxServerId` 的一个目标，通过既有 session binding 找出该进程使用过的全部 workspace directory，逐目录调用本机 `/session/status`；处理过程中持续续租，dispose 前再次对比 PID 和启动时间，防止端口复用后误操作新进程。任一 `busy/retry`、未知结构或 manager 快照缺失都递增重试次数并退避。PUBLIC scope 全部明确空闲后优先使用同一 PID/启动时间的 manager 实时快照 `sessionPath/configPath` 恢复共享配置链接，因此历史平台进程 PID 为空或启动时间源存在微小差异、target 无法映射 `userId` 时也不会永久重试；旧 manager 缺少路径时仅允许已映射用户且数据库进程身份仍精确一致的兼容路径，其他情况保持失败关闭。链接恢复后调用本机 `/global/dispose`；APPLICATION scope 只 dispose 已收敛用户，不改公共指针。某用户全部 target dispose 后立即恢复该用户；`RunApplicationService`、流式/旧版旁路问答、legacy command/shell 同时读取同一用户级闸门。
- `OpencodeProcessConfigLinkService` 管理每用户唯一的 `{sessionPath}/.testagent-runtime/current-public-config` 软链接。启动时链接 `OPENCODE_PUBLIC_CONFIG_DIR`，公共个人保存时只切到本人公共 worktree；切换使用同目录临时链接和 rename，目标若被普通文件/目录占用则失败关闭，不删除未知内容，不支持软链接的平台明确报错且绝不降级复制。`PersonalAgentConfigRuntimeReloadService` 只接受已由工作区模块校验 owner/server 的公共 worktree 配置根，要求本人 RUNNING 进程使用受管路径，切换后调用该进程原生 `/global/dispose`；该同步端口由 API 边界调度到受控阻塞线程，不能直接在 WebFlux 事件线程调用。升级前直接读取共享路径的旧进程必须先通过平台受管方式重启一次才能使用个人预览。

- Session 创建、查询、消息追加、归档和当前用户历史会话分页；用户历史由 `SessionHistoryRepository` 只读端口提供，按会话创建人、Run 触发人、消息发送人归因，保留 `pinned` 字段但排序只使用更新时间倒序。
- 定时异步执行：`NightExecutionCapacityRegistry` 实现通用内存参数 SPI，由 `CommonParameterMemoryRegistry` 在应用启动时从全局通用参数 `NIGHT_EXECUTION_SLOT_CAPACITY` 严格加载正整数容量，并在匹配的 `CommonParameterReloadedEvent` 或手工刷新时查库重载；运行中刷新失败保留上一有效值。`NightExecutionTaskApplicationService` 支持 `NIGHT_WINDOW` 和 `ADMIN_CUSTOM`：前者在北京时间 21:00 至次日 07:00 的 15 分钟时段内按容量推荐/创建/改期，后者仅允许 `SUPER_ADMIN` 选择下一完整分钟至未来 24 小时并使用 1 分钟显示区间/15 分钟重试窗口，不预留或释放夜间容量。两种模式都可绑定当前 Session 或事务内预创建空白 Session，固化目标 Linux 服务器，并以持久化锁阻断同一 Session 的普通消息发送和归档。
- 夜间 XXL 分发：`NightExecutionDispatchScanTaskHandler` 由 XXL 每分钟触发，扫描最多 500 条已到期且窗口未结束的 `SCHEDULED` 任务且不按模式过滤，按固定目标分组，每批 50 条、最多并发 8 台服务器。网关先由公共路由器选出目标服务器上的精确 backendProcessId，同服务器多 JVM 也只在选中进程执行。目标 Java 通过 `NightExecutionDispatchService` 以 `attemptId + backendProcessId + 5 分钟租约` 条件认领，最多并发 4 个普通 Run 受理调用；它不建立夜间队列，只复用用户进程初始化、会话上下文和 `RunApplicationService.startScheduledRun`，返回 runId 后即结束分发请求，不等待 Run 终态。
- 状态与补偿：`NightExecutionRunLifecycleService` 只接受来源、任务 ID、owner、Session 和 Workspace 都匹配的普通 Run 锚点，再把当前 attempt 更新为 `DISPATCHED`、关联 runId、清除完整输入并释放会话锁；仅 `NIGHT_WINDOW` 同时释放时段容量。默认 `LEGACY_FULL` Scheduled Run 也先写数据库唯一锚点，并用 Run 级 attempt/5 分钟租约/受理时间保证 anchor-only 崩溃后复用同一 runId/messageId 继续 handoff。恢复 attempt 在重投前复用普通 Run 的远端接收探测器，按 remoteSessionId + dispatchMessageId 查消息：已存在时只恢复订阅并写受理标记，明确不存在时才提交，查询为 UNKNOWN 时等待下一轮，避免 prompt 已受理而本地标记未落库造成重复执行。活动态锚点只有异步 prompt 已提交或远端消息已确认才算受理。Run 后续成功、失败或取消均不再改变定时任务。`NightExecutionDispatchLeaseGuard` 每分钟只为当前同步受理调用续租，普通 Run 创建锚点前会再以 attempt 和窗口执行一次续租 CAS。`NightExecutionDispatchOwnerWatchdog` 在 owner 本机先查锚点并收敛 handle 已消失的过期 attempt；跨 Java 的 `NightExecutionReconcileService` 每 5 分钟先按稳定 `sessionId + clientRequestId` 修复已受理锚点，无已受理锚点时仅在租约到期且精确 owner Java 心跳消失后按 attempt 恢复。窗口结束时也不会越过仍在运行的本机 handle/在线 owner 直接写失败；handle 结束后再按锚点或窗口收敛。传输至少一次，Run 唯一锚点和远端接收探测保证同一定时任务只创建一个 Run，恢复时不重复进入执行 loop。
- 会话运行上下文签发与校验；`ConversationContextApplicationService` 在所有权威读取前取得 Redis 签发租约，校验 Session owner 后先通过 `ConversationWorkspaceAccessAuthorizer` 确认托管应用已启用、用户仍是有效成员且个人 Workspace 属于本人，再通过 `TrustedWorkspaceResolver` 安全解析历史空服务器 Workspace、读取当前用户 `READY` 完整进程与 agent binding 构造控制面快照。历史 Workspace 首次回填一发生就放弃本轮保存，只允许使用全新租约重新执行一次 Session、成员、Workspace 等完整权威读取；其它 CAS 失败不重试。读取先 `peek`，校验认证用户、agent、Session、版本和过期时间后再原子 `touch`；缺失或失效分别返回 `CONVERSATION_CONTEXT_REQUIRED`、`CONVERSATION_CONTEXT_EXPIRED`，运行态存储异常返回 `RUNTIME_STATE_UNAVAILABLE`，不回退数据库或 JVM 内存。
- `ConversationRunContextResolver` 在 Run 产生数据库副作用前执行 token/兼容策略。有效 token 路径复用完整进程、Session、Workspace、ExecutionNode 和已有 AgentSessionBinding 快照，并在每次 Run 通过公共 `OpencodeProcessStatusQueryService.querySnapshot` 动态探测；该探测不按 processId 查询 Repository，稳定 `RUNNING` 为 0 次 Repository SELECT、0 次数据库写入，只有状态、PID 或服务地址确有变化时写一次。`STALE` 瞬时故障只拒绝当前 Run 并保留 token，只有明确 `NOT_STARTED` 才按进程索引失效上下文。已有远端 session 的 Run 不再重复读取其它控制面数据或重新创建远端 session。
- 主动失效入口包括：Session 归档在数据库保存前建立 revoke gate，保存失败按本次 token CAS 回滚；历史 Workspace server 回填、文件路由安全回绑、托管 root/server/status 变化、成员移除和角色调整使用覆盖关系型写入窗口的 Workspace/user mutation gate，成功时原子再次失效并释放自己的 gate token。三类 gate TTL 均为 24 小时，异常时 fail-closed 但不会永久锁死；generation 持久保留。公共进程启动/停止和明确未启动按 processId 失效；`SYS_DATA_ROOT_DIR`、应用根和个人 worktree 根的跨 Java 参数重载提升全局代次。普通 `invalidateSession` 仍是非 gate 清理。
- 用户级会话运行态摘要和状态流；用户已有 Redis 运行态 marker 时，`SessionRuntimeStateApplicationService` 从用户 active Run 索引回读 manifest，保留 `QUESTION/PERMISSION` attention 并分别计算 `questionCount/permissionCount`，active-run 也从 Session 索引读取，不再查询 PostgreSQL；未进入新链路的 legacy 用户继续使用现有只读 Repository。状态流由 `RunEventLiveBus.streamAll()` 的 run/question/permission 事件触发刷新并保留低频触发器，摘要签名包含两个待关注计数，但新链路的每次刷新仍只读 Redis，不形成数据库轮询。
- 用户级运行态 dispose 复用 `UserRuntimeDisposeCoordinator`：申请闸门时由 Redis 原子确认 `active:user` 为空，再复核全部 Session 摘要；主 Run、宠物/手册旁路问答与 legacy sideQuestion/command/shell 的新消息入口都检查同一闸门。应用 `/global/dispose` 与公共个人 `/public/runtime-reload` 共用两分钟 token 租约并每 30 秒 CAS 续租，避免其它标签页或会话在前端检查后抢先启动，也不会因 OpenCode 单次超时和重试让闸门先于 dispose 结束。
- Run 启动、取消、远端 agent session 懒创建/复用、事件订阅和终态处理；服务端自动创建 dispatch 锚点时统一调用当前 `AgentRuntime.createDispatchMessageId()`，OpenCode 因而使用原生时序 ID；`LEGACY_FULL` 的显式旧 `messageId` 仍原样透传，`REDIS_SUMMARY` 继续为新 Run 自动生成。生成后的同一个 ID 复用于远端 command、平台 USER `remoteMessageId`、root scope、manifest 和持久化锚点。平台保存 `RUNNING` 并订阅事件后异步提交远端 `prompt_async` 或原生 `/session/{sessionID}/command`，不等待远端长任务完成才返回 Run。root `run.succeeded/run.failed` 终态事件是 Run 终态事实源；两种 dispatch 的调用完成异常均作为候选失败统一等待 300ms，不依赖第三方英文错误文案，窗口内收到 root 终态时只保留根事实，无 root 终态时才追加一次安全 `run.failed`。`REDIS_SUMMARY` 在宽限期内保留原 owner lease 和事件订阅；无根终态时使用原 fencing token 写失败并投影后释放，owner 已转移或 manifest 已终态时旧执行者不得落失败。真正的事件流中断继续使用运行态丢失与 owner 重新竞争链路。后到 root 终态仍允许纠正旧服务先落的临时失败并刷新最终快照。事件订阅先经 `RunSessionScopeRouter` 判定 root/child scope，`REDIS_SUMMARY` 的 scope、durable/transient 事件与物化快照只写 Redis；部分 OpenCode 版本在 question/permission 回复后不再向既有订阅推送正文和 root idle，此时 question 回复 HTTP 成功会立即按当前 storage mode 补记既有 `question.replied` 事件，清理待回答状态并让前端回填原工具卡；回复接口还会短时轮询同一远端会话的最新消息，只有最新 assistant 明确 `finish=stop` 且平台 Run 仍非终态时，才按当前 storage mode 补发真实的 `message.updated/message.part.updated` 与 `run.succeeded`，不会把最终 assistant 文本当作 question 答案卡替代物。root 失败会结束订阅，root 成功仅在首轮标题尚待确认时继续保留该 root 的标题监听，直到原生 title agent 完成、等待被主动取消或连接断开后的补偿读取结束；订阅最终结束时统一释放 scope 缓存。
- OpenCode 内置 title agent 产生的既有 `session.updated`：仅当路由已确认当前 Run 的 root scope、远端 root ID 与平台 Session 绑定一致，且 `info.title` 有效且不是 `New session - <timestamp>` / `Child session - <timestamp>` 默认标题时，才回写对应平台 Session 标题；同时兼容 `rawPayload.properties.info.title` 包装。首轮 root Run 成功后，`RunSessionTitleWatchRegistry` 固化当次远端路由，`RunSessionTitleWatchService` 继续监听同一 root 的 title agent 完成消息，再读取最终标题并合成为既有 `session.updated`。标题写回使用 MyBatis 条件更新，只有当前标题仍是首条消息临时标题才成功，避免覆盖后到的原生标题或用户手动改名；标题在 `ACTIVE` 阶段同步成功并关闭 token 后，主 Run 仍全量路由后续 message/part/question/terminal，只有 `TITLE_WAIT/TITLE_READING` 会收窄到标题相关事件。下一轮对话、手动改名、归档/删除会取消等待。不会创建临时远端 session、二次调用 title agent 或按超时生成替代标题。本行为不新增 API、DTO、事件 wire name、数据库/Flyway 或 SDK。
- legacy stale active Run 收敛：`StaleActiveRunReconcileTaskHandler` 作为业务 `ScheduledTaskHandler` 注册到 `test-agent-scheduler`，默认 cron `0 */5 * * * *`、锁 TTL 5 分钟。MyBatis 候选查询固定排除 `REDIS_SUMMARY`，每轮最多扫描 200 条 `updated_at` 超过 2 小时且仍为 `PENDING/RUNNING/CANCELLING` 的 `LEGACY_FULL` Run；收敛前先查 Redis 运行态，`test-agent:run-output-activity:{runId}` 30 分钟内存在代表仍有用户可见输出，`test-agent:run-pending-ask:{runId}` 存在代表当前最新状态仍停在未处理 ask，二者任一存在都跳过。pending ask 只由实时 RunEvent 处理写入 Redis，不通过数据库 RunEvent 反查；当前 ask 类事件包括 `permission.asked` 和 `question.asked`，对应 `permission.replied/question.replied/question.rejected` 或 Run 终态会清理。Redis 读取异常时保守跳过，避免误杀仍在运行或等待用户处理的会话。真正超时的 legacy Run 通过 `saveIfStatus` CAS 标记为 `FAILED` 并追加数据库 `run.failed`；新模式不进入该链路。
- `REDIS_SUMMARY` 待交互 Run 由 `RunPendingAskExpiryScheduler` 在 Java 启动完成后立即扫描，并每 5 秒扫描一次当前 `linuxServerId` 的 Redis active 索引。`RunPendingAskExpiryCoordinator` 只接收 attention 非空、`attentionAt` 已达到 7 天边界且仍 active 的 manifest；执行前复用 `BackendJavaRouteResolver` 确认当前 Java 是同服务器被选中的实例，再竞争 15 秒 owner lease。未取得租约、执行器未接管或执行异常时使用原 fencing token 安全释放；实际取消和终态收敛通过 `RunPendingAskExpiryExecutor` 端口装配，缺少实现时默认 no-op，不会误取消远端会话。待交互物理 TTL 必须在 7 天业务边界后至少再保留一个 5 秒扫描窗口，生产默认使用更宽的 recovery buffer，避免 key 到期与扫描形成竞态。
- `REDIS_SUMMARY` 普通无活动 Run 由 `RunInactiveExpiryScheduler` 在 Java 启动完成后立即扫描，并每 30 秒扫描本服务器 active 索引；无 attention 且 `updatedAt` 达到两小时边界后，通过公共 Java 路由、owner lease/fencing、best-effort 远端取消与终态摘要投影收敛，不写 `run_events`。
- 新模式 Run 锚点写入后若 Redis 在远端 prompt 派发前中断，不立即把原文降级写库：启动链路先尝试 fenced 失败终态；仍不可用时由 `RunRuntimeLossConvergenceScheduler` 等待 30 秒。Redis 仍不可用则使用启动期安全控制面快照取消已创建的远端 Session（尚未创建则跳过）并写 `RUNTIME_STATE_LOST` fallback 双摘要；Redis 已恢复则以条件接管产生的新 fencing token 关闭这个从未派发的 Run，避免锚点永久停在 `RUNNING`。
- AI 整轮回复反馈归属校验和 upsert：只允许 Run 触发人或所属 Session 创建人评价主对话 `SUCCEEDED` Run，按 `(userId, runId)` 幂等更新，评论最多 300 字；旧消息入口能关联 Run 时转入该逻辑。
- 运营分析 rollup 与查询：主链路只写事实，`AnalyticsRollupTaskHandler` 以任务 key `opencode-runtime.analytics-rollup` 注册到统一 scheduler，默认 cron `0 */5 * * * *`、锁 TTL 5 分钟，刷新最近窗口的 hourly/daily rollup 和 Run 耗时直方图。scheduler 负责 Redis 分布式锁、续租、运行记录、手工触发和协作式停止；业务数据库锁暂时保留，供新旧 Java 滚动部署期间共同互斥，锁冲突只记录为本次未执行。查询服务只读 rollup 并返回 freshness，不统计、不展示、不导出 cost/costUsd。
- 当前用户 opencode 进程状态查询、头像菜单服务状态投影、初始化契约、防绕过 Run 校验、runtime 代理用户进程路由、manager WebSocket 命令网关，以及用户进程到兼容 `ExecutionNode` 的投影。
- `BackendJavaProcessLifecycleService` 首次保存后端实例时以进程启动时间作为 `createdAt`，心跳时间作为 `updatedAt`，避免启动阶段 manager 注册与周期心跳并发导致时间逆序、阻断 manager WebSocket 注册。
- `BackendJavaRouteResolver` 统一解析当前 Java 所属稳定 `linuxServerId`、Redis 中按 `backendProcessId` 保留的 Java 快照、按 `backendProcessId` 精确选择一个在线 Java，以及 `containerId` 对应 manager 所属服务器；同一服务器多 Java 的服务器级命令优先选择与目标服务器 manager 已连接的 Java，其次选择同服务器最新心跳 Java，最后才使用当前 Java 兜底。首次分配选服会对每个 `containerId` 只保留最新 manager 快照，汇总服务器全部 manager CONNECTED 容器的 `currentProcesses`（已满容器也计入），并要求至少一个 READY、未满且与所选 Java CONNECTED 的容器；结果按总进程数、`linuxServerId` 稳定排序。JVM 内存参数等进程级诊断必须使用精确进程方法，不合并同服务器 Java。所有用户进程、运行管理、Agent 配置和文件 WebSocket 路由都必须复用它做目标选择。新增 opencode-manager 路由或 Java->manager 控制入口时，不得在其它 service 中重新扫描 Redis 快照、复制 `linuxServerId/containerId` 解析、直接控制远端 manager 或绕过目标 Java。
- `RunEventSseRouteService` 优先从 Redis Run manifest 读取创建时固定的 `producerLinuxServerId` 并通过公共 `BackendJavaRouteResolver` 选择目标 Java，不读取当前用户 binding，也不查询 routing/process 表；manifest 缺失的 legacy/旧 Run 才按 `routing_decisions -> node_ocp_* -> opencode process` 兼容解析。SSE 使用可回退入口，目标不可用时返回空供 API 层本机只读恢复；cancel 等写操作使用严格入口，归属缺失、节点无法映射或目标 Java 不在线时直接失败，禁止在非 owner Java 执行副作用。
- `OpencodeProcessStatusQueryService` 是唯一公共 opencode server 状态查询封装。强状态查询先确认平台进程记录是否存在，再通过本服务器 manager health 归一为 `NOT_STARTED/RUNNING/HEALTH_CHECK_FAILED/STALE`，并按健康结果回写数据库与 Redis heartbeat；`querySnapshotReadOnly` 供启动竞争补偿和 tracked 停止前置探测等旧坐标精确确认使用，返回 manager 最新 PID 的规范快照但不写数据库、不刷新 heartbeat；manager 报 `HEALTHY` 却缺少有效 PID 时必须 fail closed 为 `STALE`，不得回退数据库旧 PID。弱健康检查 `weakHealth` 只读取 Redis manager snapshot，直接调用 opencode `/global/health`，同样不读写数据库、不调用 manager gateway、不刷新 heartbeat，供前端 10 秒健康轮询使用。新增用户状态、Run 前置检查、运行管理探测、启动后确认、停止后确认或后台心跳刷新时必须复用它，不得在业务 service 中直接调用 `OpencodeProcessManagerGateway.checkHealth()` 后自行映射状态。
- `OpencodeProcessStartupService` 是唯一公共 opencode server 启动封装，负责启动前把用户固定公共配置软链接恢复到共享运行副本，再由目标 Java 调用本服务器 manager `start`，保存候选进程、在 manager command-timeout 窗口内用只读公共状态探测等待本地 state/PID 和 opencode HTTP health ready，再以 process/assignment 坐标及旧 `status/PID/traceId` 生命周期代次 CAS 显式回写 `RUNNING/STOPPED/UNHEALTHY/FAILED`、Redis heartbeat、ACTIVE binding 和兼容 `ExecutionNode`，避免同坐标新实例被迟到结果覆盖。启动回包的 `processCreated` 使用三态语义：新 manager 新建进程为 `true`、幂等复用为 `false`、旧 manager 缺字段为 `null`；只有 `true` 且候选或最终状态 CAS 冲突时，才通过公共停止服务按本次 start 的 UCID+PID 精确补偿，`false/null` 绝不清理可能复用的既有实例。启动命令显式携带稳定 `configPath`，并优先从用户仓储解析统一认证号作为可选 `unifiedAuthId` 下发；已有 ACTIVE binding 的精确恢复还携带可选 `bindingRecovery=true`，使新版 manager 不把恢复误判为新增容量占用，首次分配和端口迁移保持 `false`。只有无法读取用户记录时才从已校验的 `.../users/{unifiedAuthId}` session 路径派生，绝不使用平台 `userId` 冒充。manager 据此为每次启动创建 `{safeUnifiedAuthId}-{UTC 启动时间}-{port}.log`，同时设置 `OPENCODE_CONFIG_DIR` 并拒绝把配置路径不同的健康旧进程冒充为本次启动结果。滚动升级兼容旧 Java/旧 manager：可选字段会被旧 manager 忽略，新 manager 能从稳定 session 路径补齐；既有 `{port}.log` 不迁移。启动环境会合并调用方值，并在调用方未提供时按目标 Java 当前平台读取 `common_parameters.OPENCODE_REFERENCES_DIR` 注入用户进程；滚动升级时参数缺失不阻断启动，既有运行进程不自动重启，只有新启动或走公共停止/启动程序的受管重启获得新值。新增初始化、重启后拉起、端口复用或其它启动入口时必须复用它，不得在业务 service 中直接调用 `OpencodeProcessManagerGateway.startProcess()` 后自行写状态。
- Go manager 收到公共启动服务下发的 `sessionPath={OPENCODE_SESSION_DIR}/users/{unifiedAuthId}` 后，先合并 `command.environment`，再保留控制权并覆盖 `HOME=sessionPath`、`XDG_DATA_HOME=sessionPath`、`XDG_CACHE_HOME=sessionPath/.cache`、`XDG_STATE_HOME=sessionPath/.local/state`、`TMPDIR=sessionPath/.tmp` 和显式 `OPENCODE_CONFIG_DIR`。Linux fork 前会把 session、cache、state、tmp 创建为 `0755` 普通物理目录；文件或软链接冲突直接启动失败，目录在停止与重启后持续保留。公共配置仍使用 Java 维护的软链接；新增环境只对新启动或公共服务受管重启后的进程生效，同一 manager 系统账号下仅提供目录级隔离。
- 用户进程初始化支持可选 `operationId` 进度记录。`UserOpencodeProcessAssignmentService` 负责校验请求、确认分配、选择容器和准备参数等分配层步骤；候选统一由 `LiveOpencodeContainerCandidateResolver` 读取 TTL 10 秒的 Redis manager 快照，同容器只取最新心跳，并要求 manager `CONNECTED`、容器 `READY` 且未满、快照连接列表包含当前 `backendProcessId`、当前 Java 内存中仍持有该容器 manager WebSocket。Redis 访问异常返回 `RUNTIME_STATE_UNAVAILABLE` 且不回退数据库；正常无候选返回 `OPENCODE_UNAVAILABLE`。`OpencodeProcessStartupService` 负责进程启动、记录候选进程、检查进程、健康检查和写入绑定等公共启动步骤；失败时通过 `OpencodeProcessStartOperationRepository` 保存 `errorCode/errorMessage/traceId`，查询进度只读数据库，不触发 manager health/start 或 RunEvent。
- `OpencodeProcessReservationService` 是初始化和端口迁移的短事务预留入口。事务内固定先锁用户、再锁目标 Linux 服务器，首次分配先写 `STARTING` process 与 `ACTIVE` binding，端口迁移则以当前 `processId + linuxServerId + port + status/PID/traceId` 代次同时 CAS process/binding；manager 调用始终发生在事务提交后。并发初始化发现已有 binding 时只沿用胜出的 processId/端口，不再创建第二个绑定或端口；跟随请求只读取权威数据库状态，胜者仍为 `STARTING` 时返回“初始化正在进行”，不得再次发送 manager start/stop 或执行冲突补偿。首次分配和明确端口迁移继续使用原有候选容器、容量排序和首个空闲端口规则；空闲集合在数据库历史占用端口之外补充 Redis manager 实时进程端口，避免选中仍存活但平台未登记的无主端口。
- `OpencodeProcessStopService` 是唯一公共 opencode server 停止封装。平台已有进程记录的 tracked 停止先校验请求快照与数据库 process/assignment 坐标及旧 `status/PID/traceId` 生命周期代次一致，再以该持久化代次做只读 manager preflight 取得最新 PID，并在副作用前再次校验代次；随后从 `UserRepository` 读取权威统一认证号，只发送 `stopOwned(expectedUnifiedAuthId, expectedPid)`。manager 在同一生命周期锁内同时匹配端口、UCID 和 PID 后才允许 signal/delete，任何状态不确定、身份/PID 缺失、旧 manager 不支持或 ownership mismatch 都 fail closed，绝不回退端口 `stop`。停止成功后继续用原持久化代次调用公共状态查询确认未运行，并以同一代次 CAS 回写 `STOPPED`；manager 已明确无进程时也会通过普通状态查询完成该回写。只有没有平台进程记录的无主 manager state 才保留 legacy 端口 `stop/restart` 并仅信任 manager 回包。UCID+PID 无法区分极窄窗口内同一 UCID、同一 PID 的新代次（包括 PID 数值复用或同实例并发接管）；若要彻底消除该残余风险，协议还需引入不可复用的 incarnation token。新增运行管理、清理、回收或其它停止入口时必须复用本服务，不得在业务 service 中直接调用 gateway 后自行判定或写状态。
- `WorkspaceFileRoutingService` 根据当前用户 opencode 进程的服务器归属和统一 Java 后端解析器定位同服务器后端 Java 进程，供前端工作区文件 WebSocket 先路由到目标后端；路由前复用 `ConversationWorkspaceAccessAuthorizer` 校验当前用户仍是工作区所属应用的有效成员，`SUPER_ADMIN` 不旁路托管成员关系，非托管 Workspace 仅向 `SUPER_ADMIN` 服务器工作空间兼容入口开放。该文件路由归属查询只读取 ACTIVE binding 和可恢复进程记录，不触发 manager health/start 命令，避免文件树加载被 opencode-manager 慢响应阻塞。超级管理员服务器工作空间选择器也复用该在线快照返回活跃后端服务器列表和默认目录。本地服务器身份变化或切换数据库后，历史 workspace 的 `linux_server_id` 若指向已无在线 Java 后端的旧服务器，且当前用户 opencode 已迁移到本后端、workspace 根目录在本机可访问，路由时会把 workspace 回绑到当前 `linuxServerId`；旧服务器仍在线或本机目录不可访问时继续返回 `CONFLICT`。ticket 签发与每条 workspace RPC 还会再次校验成员关系，避免已签发 ticket 在成员撤权后继续读取工作区或引用内容。
- `AgentRuntimeTargetResolver` 统一封装用户进程节点、固定节点 fallback、远端 session 创建/复用以及 binding 节点不一致时的自动覆盖。创建或重建根远端会话时不传平台临时标题，保留 OpenCode 默认标题以触发内置 title agent；平台仍在收到并确认 `session.updated` 后同步生成的标题。复用已有 binding 前会调用 agent runtime 校验远端 session 是否仍存在；opencode 返回 404 时会记录结构化 WARN 并创建新的远端 session，同时更新通用 `AgentSessionBinding` 和兼容 `sessions.opencode_*` 字段，其它远端错误不吞掉。
- `SideQuestionStreamingApplicationService` 创建从一开始即归档的内部 Session 与 `SIDE_QUESTION` Run，复用目标路由和临时 fork：fork 直接继承主对话上下文，本次 prompt 只携带用户问题、可选模型和 `tools={"*":false}`，不再触发 build agent、只读工具链或额外 compact；发送前快照 fork 已继承的 message ID，发送后只投影新增 assistant，不能依赖 OpenCode fork 后仍可能指向历史 assistant 的 `parentID`，文本增量继续投影到既有 RunEvent SSE。远端漏发 idle/finish 时每 3 秒低频读取临时会话消息恢复终态，完成后删除 fork，最终答案上限 64 KiB。旧 `OpencodeRuntimeApplicationService.sideQuestion` 同步入口继续兼容，主 Session 不写入旁路问题或回答。
- `SideQuestionOrphanCleanupTaskHandler` 每 5 分钟扫描超过 10 分钟仍 active 的旁路 Run，按归档内部 Session 的持久化远端映射在原节点幂等删除 fork，再复用 `SideQuestionTerminalService` 事务 CAS 追加唯一 `run.failed`。无映射时平台 Run 仍收敛并记录潜在远端泄漏窗口；其它节点/删除故障留待下轮重试。
- `RuntimeManagementQueryService` 聚合 Linux 服务器、后端 Java 进程、opencode 容器、manager、manager 管理的本地 opencode server 明细、manager-backend 连接、用户进程和绑定状态，供超级管理员运行管理页展示；Java 后端、manager 和连接在线态只读取 Redis 快照，不回退数据库 heartbeat 字段。manager 明细中的 `startCommand`、`unifiedAuthId` 和 `managerStatus` 保持可空以兼容旧快照，并按同服务器、同容器、同端口扫描候选用户进程，再结合 `ACTIVE` 用户绑定标记 `BOUND/UNBOUND` 归属；该归属不依赖页面底部用户进程分页筛选，也不从启动命令反向解析 UCID。无平台记录的 manager 进程保持 `UNBOUND`，仅展示“平台未登记”和“未执行 HTTP 健康检查”，不会自动认领、停止或改绑。底部用户进程查询通过用户关键字定位用户名、`userId` 或统一认证号，不依赖 Redis 进程心跳过滤，读取数据库历史进程后只对当前服务器进程通过公共状态查询服务探测；远端服务器进程返回 `REMOTE_SERVER/CHECK_SKIPPED`，避免随机 Java 直接控制其他服务器 manager。容器指标历史按容器 ID 读取；后端 Java 指标历史主接口按稳定 `linuxServerId` 读取，把同一服务器下服务器 CPU/load/内存/swap/磁盘样本与 Java 进程/JVM 样本合并返回，保留近 48 小时原始 5 秒样本，查询时按 Controller 解析出的时间窗口读取，超出查询上限时按时间桶降采样；旧 `backendProcessId` 指标入口仅兼容旧客户端，能解析到稳定服务器身份时委托同一服务器历史。
- `RuntimeManagementCommandService` 为超级管理员运行管理页提供按 `containerId + port` 重启/停止 opencode server 的命令入口，复用公共启动/停止服务转发到 manager WebSocket，不直接访问 opencode server；跨 Java 后端路由由 `test-agent-api` 的运行管理路由服务先按统一 resolver 定位容器所属服务器，再交给目标 Java 调用本服务。对于平台已有记录的用户进程，`RUNNING/UNHEALTHY/STARTING` 等非 `STOPPED` 状态会先调用 `OpencodeProcessStopService` 以 UCID+manager 最新 PID 执行 owned stop，再按原 `process.sessionPath`、`containerId + port` 调用 `OpencodeProcessStartupService` 重新拉起并做启动后 health 确认；`STOPPED` 记录直接走公共启动服务。没有平台用户进程记录的无主 manager state 才保持原 manager `restart/stop` 端口语义，不写进程/binding。
- `OpencodeProcessHeartbeatMaintenanceService` 每 3 分钟只扫描当前服务器的 RUNNING opencode server 进程，通过公共状态查询服务确认并刷新 Redis 心跳，每 5 分钟清理 Redis 心跳索引中过期的 Java/opencode 进程 ID；opencode 进程心跳仍保留 5 分钟窗口。
- `BackendJavaProcessLifecycleService.registerHeartbeat` 每 5 秒为当前 Java 实例按 `backendProcessId` 写 Redis 后端快照，TTL 为 10 秒，并通过 `BackendRuntimeMetricsCollector` 采集服务器、Java 进程和 JVM 指标；Linux 环境使用 `/proc/stat` 两次差值、`/proc/loadavg`、`/proc/meminfo`、`/proc/self/status`、`/proc/self/fd` 和 JVM MXBean，非 Linux 保持原 best-effort CPU/内存/磁盘口径且 Linux-only 字段为 `null`。运行管理磁盘容量默认监控通用参数 `SYS_DATA_ROOT_DIR` 解析后的路径，缺失或非法时回退 Java 当前工作目录；不读取新环境变量。在线服务器身份、服务器级样本和 Java/JVM 样本都按稳定 `linuxServerId` 保存，使同一服务器 Java 进程重启后指标趋势连续。`backendProcessId` 继续作为当前 Java 实例元数据、manager-backend 连接和拓扑连线字段，容量摘要包含当前 `advertisedHost`。数据库只在拓扑首次落库或状态变化时写入 `linux_servers` / `backend_java_processes`，并保留 manager/backend 连接历史；这些数据库状态不再参与用户进程候选在线与容量判断。
- Java 心跳快照额外携带 Spring Boot build-info 构建时刻格式化出的 `buildVersion`，manager 心跳快照透传 Go linker 注入的同名字段，格式统一为北京时间 `VyyyyMMdd.HHmmss`。`RuntimeManagementQueryService` 只从当前 Redis 快照透传版本，不跨服务器探测、不以启动时间补值；旧 Redis JSON、旧 manager 和旧构造调用缺字段时保持 `null`。
- `ManagerControlApplicationService.register` 会在写 manager-backend 连接前先补齐当前 Java 后端进程拓扑，避免 Web 端口已监听但 `ApplicationRunner` 首次心跳尚未落库时触发 `backend_java_processes` 外键失败；持久拓扑写入仍是 best-effort，`opencode_container_managers` 等历史拓扑表的唯一键冲突不会阻断 WebSocket 注册。manager 启动和在线态以控制面连接及 Redis 快照为准，避免已过期的数据库拓扑行导致 manager 重连失败。
- `OpencodeProcessManagerGateway` 生产装配只有 `SocketOpencodeProcessManagerGateway`，统一通过本服务器 manager WebSocket 控制面执行 `start`/`health`/`restart`/`stop`/`stopOwned`；tracked 业务只允许 `stopOwned`，旧 manager 返回未知命令时直接映射固定网关错误且不透传内部消息、不回退 `stop`。本地开发也必须启动 Go manager，不再支持 `gateway-mode=local` 或本地直连 `baseUrl` 绕过。
- manager WebSocket 控制面通过 `ManagerControlMessageCodec` 编码 JSON；发给 Go manager 的时间字段必须保持 RFC3339 字符串，不能使用 Jackson 默认时间戳，否则 Go `time.Time` 解码会断开连接并导致用户进程初始化不可用。
- `UserOpencodeProcessAssignmentService` 不再支持 local-direct 合成进程；`status`、`fileRoutingAffinity` 的可初始化判断和 `initialize` 都复用 Redis 实时候选解析，`requireReadyProcess` 继续走 ACTIVE binding 和公共状态查询。`initialize` 按实时 `currentProcesses`、`containerId` 排序逐个候选查询数据库空闲端口；真实启动、候选进程保存、启动后状态查询、binding、heartbeat 和兼容节点投影全部委托 `OpencodeProcessStartupService`。只有本地连接表在命令发送前已断开时才安全尝试下一候选；命令超时、发送后失败、容量/配置错误和健康检查失败均立即终止，避免重复启动。瞬时 `STALE` 仅在最近一次成功健康检查后的 60 秒内沿用 READY，超过宽限期后状态页和 Run 前置校验都会拒绝继续使用旧绿灯。
- `UserOpencodeProcessAssignmentService` 对已有 binding 的用户只允许在原 `linux_server_id` 上重建；当请求落到其他 Java 后端时，由 API 层先按 ACTIVE binding 的 `linuxServerId` 路由到目标服务器 Java，再由目标 Java 控制本服务器 manager，不允许当前 Java fallback 到其他服务器并迁移 binding。是否已分配只读取 `user_opencode_process_bindings` 的 ACTIVE 记录；`allocationStatus` 是状态查询降级专用只读入口，不触发 manager health/start 或容器可用性检查，目标后端不可用时仍返回 `linuxServerId/port` 让前端展示稳定服务器身份，只有能解析到当前在线 Java 地址时才返回 `serviceAddress`。端口选择按数据库唯一约束 `(linux_server_id, port)` 在同服务器全局避让所有历史进程行，包含其它容器和非运行态脏数据。
- 已有 binding 恢复时直接使用数据库记录的服务器、容器和端口，不参与空闲端口扫描，也不受容器容量过滤影响：`RUNNING` 直接返回，`NOT_STARTED` 在原端口启动，`HEALTH_CHECK_FAILED` 先经公共停止服务确认退出再在原端口启动，`STALE` 保留 binding 并返回状态暂不可确认。只有 manager 明确返回 `PORT_CONFLICT` 或 `PORT_OUT_OF_RANGE` 才允许在原服务器按既有规则更换端口；`IDENTITY_ALREADY_MANAGED`、`IDENTITY_CONFIG_MISMATCH`、配置/容量错误、manager 断连、超时和普通启动失败均保留原 binding。滚动升级期间旧 manager 的未知错误同样 fail closed，不迁移端口。
- `UserOpencodeProcessAssignmentService.fileRoutingAffinity` 是文件 WebSocket 路由专用只读入口，只表达文件操作应落到哪台 Linux 服务器，不代表 opencode server 健康可用；Run、初始化、头像状态仍调用 `status` / `initialize` / `requireReadyProcess` 保持强健康检查语义。
- 进程 assignment 迁移通过同一 MyBatis 原子端口同时更新 process 与 binding；process 的 CAS 除旧 server/container/port 坐标外还匹配旧 `status/PID/traceId` 生命周期代次。越界旧端口完成精确停止后，迁移前必须重读 ACTIVE binding 与 process，并确认坐标未变、状态为 `STOPPED`、PID 已清空且 traceId 属于本次停止；任一不一致都 fail closed，不能把同坐标并发拉起的新生命周期迁移到其它端口。
- `UserOpencodeProcessAssignmentService` 创建用户 opencode 进程时，session/config 路径读取 `common_parameters.OPENCODE_SESSION_DIR` 和 `common_parameters.OPENCODE_PUBLIC_CONFIG_DIR`；缺失或空白时抛平台错误，不回退环境变量或代码默认路径。opencode 原生 session 数据目录固定为 `{OPENCODE_SESSION_DIR}/users/{unifiedAuthId}`，Java 通过用户仓储解析统一认证号，并拒绝空白、`/`、`\` 和 `..` 等无法作为安全路径片段的统一认证号；旧 `{OPENCODE_SESSION_DIR}/{port}` 目录不自动合并。未绑定初始化由 API 层先选定集群中进程总数最少的可初始化服务器；请求到达目标 Java 后，再按 Redis manager 最新快照中的实时进程数选择有空闲端口的本地目标容器，然后经公共启动服务向该容器对应的 manager 下发携带 `sessionPath` 的 `start`；manager 使用已通过 `configUpdate` 同步的配置路径。`OPENCODE_PUBLIC_CONFIG_DIR` 是否存在且非空只由目标 manager 在所在服务器检查，Java 本机不做 `Files.*` 校验。manager 返回 `errorCode=OPENCODE_UNAVAILABLE` 时，message 会包含目标服务器和 manager 实际检查的配置目录，socket gateway 映射为同码平台错误并原样透出。manager `STARTED` 仅表示启动命令完成，公共启动服务还必须立即调用 health；health healthy 才能返回 READY。
  `baseUrl/serviceAddress` 使用当前服务器 advertised host 和端口生成，不使用 `linuxServerId` 拼接地址。
- RunEvent 持久化策略、实时发布和 agent projected messages 恢复；Run 启动后记录 root session scope，`RunSessionScopeRouter` 根据 task/tool metadata、`session.created/session.updated parentID` 和 `session.children` bootstrap 发现 child/grandchild；task metadata 指向新 session 时只发布 discovery/scope updated 供前端建立导航索引，原始 `message.part.updated` task part 仍归属于发起它的 root 或 child scope。原生 opencode 先发 pending task part、再发 child `session.created(parentID)` 时，router 会在同一 parent 下用 FIFO pending task 队列补齐 `taskMessageId/taskPartId/taskCallId`，并从 child `info.agent/info.title` 提取展示 metadata；同时过滤 `heartbeat/server.heartbeat/tui/pty/workspace/worktree/installation/plugin/catalog` 等无 session 归属的全局 unknown 噪声。router 为每个订阅维护 known sessions 与 `scopeVersion`，稳定事件不反查数据库；`REDIS_SUMMARY` 完全不读写 `run_session_scopes`，只使用订阅态和 Redis cache，legacy 仅在 cache miss/发现新 child 时兼容读写数据库。终态调用 `finishRun` 释放本机订阅态。
- Run 运行数据面按 manifest 固定 `LEGACY_FULL` 或 `REDIS_SUMMARY`，活动 Run 禁止中途切换。新模式的 durable/transient 事件、scope、dedup、pending 和 active 索引只通过 `RunRuntimeStore` 读写 Redis；SSE 首帧总发送 Hash/ZSET 物化 snapshot reset，随后最短 5 秒安全扫描/live 即时唤醒只按 runtimeVersion 分页读取 Redis 全事件 Stream，不做 PostgreSQL 轮询。容量换代后的 snapshot 必须仍含本轮 USER 输入、最新 assistant message、对应最终可见 text part 与 run-status；tool/reasoning part 或非 assistant message 不得占用这些关键保留位。`RunStorageModeSelector` 只对携带已校验 `contextToken + clientRequestId` 的新客户端按 userId 稳定哈希灰度；默认开关关闭、比例为 0，回滚只影响后续新 Run。
- Redis Run owner lease 为 15 秒、每 5 秒续租，fencing token 在单 Run 内单调递增。续租使用独立单线程调度器，恢复、到期收敛和终态重试使用另一组 4 线程维护调度器，禁止远端 I/O 或批量数据库重试饿死 lease。续租返回空值表示 fencing 已被其它 owner 取得，`ownership.lost()` 正常完成并只停止旧订阅；续租抛出 `RUNTIME_STATE_UNAVAILABLE` 等异常时，`ownership.lost()` 保留原异常结束，使启动订阅和恢复订阅进入 30 秒运行态丢失收敛，禁止把 Redis 故障误当成正常接管。恢复、两类到期扫描和用户取消都使用 `claimOwnerLeaseIfUnchanged` 在 Redis Lua 内原子确认 manifest 未变化并提升 token；stream append、transient 投影、远端 Session 回填以及 scope/dedup/pending 写入同时校验 owner + token，旧执行者不会在接管后刷新索引或覆盖终态。Java 启动及周期扫描先读取本服务器 active 索引，公共 `BackendJavaRouteResolver` 选中的同服务器 Java 才能接管；只有失去本机 owner 的故障候选会执行一次 PostgreSQL 控制面锚点检查：数据库已终态则 fenced 收敛 Redis，初始化后 30 秒仍无锚点则清理未派发孤儿 manifest，正常 Run 启动和事件链路不增加查询。接管再以单次 10 秒超时从 OpenCode 会话尾部按 `dispatchMessageId` 确认是否接收：`ACCEPTED` 只恢复 SSE，`NOT_ACCEPTED` 用新 fencing token 写安全失败终态，`UNKNOWN` 等待下轮，三者都禁止重发 prompt。
- `RunConversationSummarizer` 为 `REDIS_SUMMARY` 终态投影生成固定 USER/ASSISTANT 双摘要：USER 最多 512、ASSISTANT 最多 2000 个 Unicode code point，确定性移除 `<context>`、reasoning、工具输入输出、附件正文、data URL、控制字符和常见密钥模式；超长标记 `PARTIAL`，空内容或清洗异常只返回不含原文的 `FALLBACK`。每条结果固定携带 `contentKind=SUMMARY`、摘要版本和角色，供下游终态事务批量写入最多两条历史消息。
- `RunMessageRecoveryService` 为 Run/Session 历史入口固定执行 Redis 24 小时物化详情 → OpenCode 会话 → PostgreSQL 双摘要降级。Run 级 OpenCode 恢复按平台 USER、root scope 或 `RunDetailsLocator.dispatchMessageId` 的稳定锚点裁剪，只选择该 user 及 `parentID/parentId` 直接指向它的 assistant；锚点未到达、来源冲突、重复 cursor、20 页超限或旧 Run 时间窗内存在多个 user 时返回空投影。已记录 child scope 可独立恢复，但只有选中的 root 消息能发现新 child。Session 级历史仍保留完整多轮 user/assistant 会话。Redis 命中不读取关系库控制面或旧事件表；摘要会映射成兼容 reducer 的 message/part 事件并标识 `SUMMARY`。legacy SSE 初始快照只回放目标 Run 的 assistant，避免与平台已有 user 消息重复并阻止旧轮 Todo 回灌。
- `RunTerminalProjectionService` 在新模式 root 终态后只读 Redis input/物化快照，以一个事务 CAS 更新无原文 Run 锚点、批量写最多两条摘要并刷新 Session 时间；终态事件 Lua 原子发布 versioned outbox，恢复协调器仍在启动和安全恢复扫描中补做幂等 CAS。数据库异常只把已清洗投影写入独立 Redis 重试存储。`RunTerminalProjectionRetryTaskHandler` 由 XXL 任务 `opencode-runtime.terminal-projection-retry` 每 5 秒触发，本模块不再启动本地 retry ticker；handler 按固定退避重试并 ack 对应 outbox version，最长保留 24 小时。
- `RunRuntimeLossConvergenceService` 处理 Redis 运行态连续中断：调用方等待固定 30 秒 grace 后传入启动时缓存的安全控制面快照，服务先复查 manifest；仍不可用时在 10 秒边界内 best-effort 取消远端会话，再直接写 `RUNTIME_STATE_LOST` 的 `FAILED` 终态和固定 `FALLBACK` 双摘要。该链路不读取 Redis input/Stream/snapshot；数据库失败时只把该安全投影写入 `TERMINAL_PENDING_DB` 重试队列。若 PostgreSQL 与 Redis 同时不可写，调度器在 JVM 内按 5 秒、15 秒、30 秒、1 分钟、2 分钟、5 分钟退避并以 5 分钟封顶持续重试，直至任一存储接受安全终态；禁止把原文降级写入 PostgreSQL。
- 只有 `LEGACY_FULL` 在 Run 终态/取消后执行 `session_messages` 快照持久化：先完整收集并按同一稳定 USER 锚点选择当前轮，再仅 upsert 本轮 assistant 可见 text、完整 message parts 和最后一条 assistant 的 token/cost；无法证明归属时不写任何消息。普通 Session 消息刷新继续同步全会话，但只保留已有 `runId`，新发现的历史 assistant 写空 `runId`，不得重新分配旧轮归属。`REDIS_SUMMARY` 不调用该快照链路，也不写原始 parts。
- 从完成态 `write`/`edit`/`apply_patch` tool part 派生运行中 `diff.proposed`，供前端实时追踪文件变化；不调用 opencode `/vcs/diff?mode=working`，实际 Git patch 和精确行数由工作区 Git Diff 接口读取。
- Run Diff 查询、接受和拒绝；`REDIS_SUMMARY` 优先读取 Redis 物化 Diff，只有远端 message/part 定位缺失时才读取 Run 非原文锚点，manifest 已过期时返回 `RUN_DETAILS_EXPIRED`，不回退 legacy `run_events`。接受/拒绝先追加 Redis action，再各用一条关系库 UPDATE 增加 Run Diff 计数，不写新模式原始事件。
- agent runtime 能力映射，包括 catalog/fs/vcs/lsp/mcp、config、provider auth/OAuth、worktree、session share、permission/question 和 MCP auth；opencode 原路径作为当前标准适配形态。
- Model/Provider 目录编排：前端对话框始终通过 runtime 代理 opencode 原生 `/api/model`、`/api/provider`，不再从 `ai_model_configs` 或 `ModelCatalogApplicationService` 返回托管目录；Run 启动前不再 `PATCH /global/config` 同步 provider。
- 内部模型代理：按 `X-Enterprise-Model-Provider` 从同一代 JVM 不可变快照同时查供应商地址和该 Provider 关联的 Token，向上游注入对应 Token 与 `ucid`，并把流式 `<think>...</think>` 转换为 `reasoning_content`；单次请求不访问数据库。
- PTY terminal ticket、限流、JVM 内 active registry、Pty4J 进程适配、真实 resize 和审计；同一链路同时承载 workspace shell 与默认关闭的服务器 shell。workspace shell 仅允许 `sh`、`bash`、`zsh`；服务器 shell 固定 `/bin/bash`、固定工作目录和不含 Java 密钥的最小环境，操作系统用户与权限直接继承目标 Java 进程，不切换用户或提权。服务器 shell 通过 jar 内置、运行时释放的临时 rcfile 提供绿色用户/主机、蓝色目录提示符，以及 `ls`、`grep`、`git` 的交互式 ANSI 配色；不会写入用户 `.bashrc` 或全局 Git 配置，用户现有 `.bashrc` 最后加载并可覆盖平台默认值。

## Model 目录配置

`test-agent.model-catalog.source` 和 `ai_model_configs` 相关类保留历史兼容，但不再参与前端模型目录、供应商目录、Run 模型校验或默认模型回退。内部供应商维护在 `internal_model_providers`，外部 Token 记录在 `internal_model_tokens` 并由 `token_id` 关联；旧 `internal_model_proxy_settings` 只供滚动升级兼容。Java 启动和既有刷新事件会用一次联表查询构建 `InternalModelProviderRegistry` 的 `providersById` 与 `authTokensByProviderId`，地址和 Token 在刷新瞬间不会串代。

## 后续 AI 编码指引

新增与会话、运行、事件、Diff、permission/question、runtime catalog、terminal 相关业务编排时改这里；新增 agent 适配器应放在 `test-agent-agent-runtime`。Controller 和 URL 映射必须放在 `test-agent-api`。
`LEGACY_FULL` 下高频文本 delta、message projection 和大段 tool/bash 输出不应写入 `run_events`；消息内容刷新恢复优先从 agent 标准 session messages 分页拉取并 upsert 到 `session_messages`，兼容消息列表接口的远端刷新必须在 bounded-elastic 线程执行，agent 不可用时回退数据库快照。`REDIS_SUMMARY` 下所有运行中 mapped/transient/durable 事件都进入 Redis，不写 `run_events` 或 `run_session_scopes`；durable 使用 seq Stream，所有事实帧使用 runtimeVersion Stream，消息/part/entity 当前状态使用 Hash/ZSET 物化，SSE 首帧 reset 后从 runtimeVersion tail 顺序恢复。不得为了 Redis 故障把原始详情降级写数据库。child idle/error 只能产生 session 事件，root idle/error 才能派生 `run.succeeded/run.failed`。
运营分析新增指标时优先扩展 `AnalyticsModels`、`AnalyticsRepository`、`AnalyticsRollupApplicationService` 和查询服务；调度入口必须继续使用 `AnalyticsRollupTaskHandler`，不得重新增加业务 `@Scheduled`。API 查询不得绕过 rollup 直接扫原始事实表，导出字段不得包含 prompt/assistant 原文、密钥或费用字段。
生产 `OpencodeProcessManagerGateway` 通过 manager WebSocket 控制面下发 `start`/`health`/`restart`/`stop`/`stopOwned` 命令；tracked 停止必须使用 UCID+PID 的 `stopOwned` 且不得回退，只有无平台记录的管理员操作可以使用端口 `stop/restart`。无连接、超时或异常必须转换为平台 opencode 错误码。测试仍可使用 fake gateway 固定初始化、健康检查或运行管理命令结果。
所有 opencode server 启动入口必须调用 `OpencodeProcessStartupService`，由公共服务统一完成启动、候选进程快照、启动后 health 和最终状态回写；不要在新增业务编排中直接调用 `gateway.startProcess()` 并自行写进程、binding、heartbeat 或 `ExecutionNode`。
所有 opencode server 停止入口必须调用 `OpencodeProcessStopService`，由公共服务统一完成 tracked 精确预检、manager owned stop、停止后 health 失败确认和最终状态回写；不要在新增业务编排中直接调用 `gateway.stopProcess()` / `gateway.stopOwnedProcess()` 并自行写 `STOPPED`。
所有 opencode server 状态查询入口必须调用 `OpencodeProcessStatusQueryService`。强状态查询由公共服务统一完成进程存在性判断、manager health、状态回写和 Redis heartbeat；弱健康查询只能走 `weakHealth`，保持只读 Redis 快照和直接 opencode `/global/health`。不要在新增业务编排中直接调用 `gateway.checkHealth()` 并自行写 `RUNNING/STOPPED/UNHEALTHY/FAILED`。
`OpencodeManagerConfigSyncService` 把通用参数表中的 `OPENCODE_MANAGER_MAX_PROCESSES`（`platform=all`）、`OPENCODE_SESSION_DIR`、`OPENCODE_PUBLIC_CONFIG_DIR` 经控制面 WebSocket 下发给已连接 manager：Java 收到 `register` 后只返回 `registered`，manager 收到后发送一次 `configRequest` 拉取完整配置；连接断开后 manager 无限重连，重连成功后重新拉取。前端只允许修改最大进程数和公共 Git 地址 `OPENCODE_PUBLIC_AGENT_GIT_URL`（均为 `editable=true`），参数刷新事件只广播 max-only `configUpdate` 给当前 Java 持有的本服务器 manager；路径类参数属于部署/初始化参数，不通过前端热刷新。公共 Git 地址虽可在前端修改，但不触发 manager 热刷新，其更新经 `common-parameter.refresh-requested` 广播保证跨实例 DB 一致后，由 AgentConfig 在下次公共配置操作时按当前部署模式直读 DB 生效。参数缺失、空白或最大进程数非正整数时不下发可启动配置，manager 保持未 ready 并拒绝启动用户进程；`opencode_containers.max_processes` 仍由 manager heartbeat 回报的生效值同步。
runtime 代理入口有认证用户时必须通过 `AgentRuntimeTargetResolver` 使用用户专属 opencode 进程；无用户主体的 static-token 或本地兼容调用才允许使用固定 `execution_nodes` fallback。
