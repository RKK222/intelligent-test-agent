# 应用工作区与版本管理需求设计文档

**版本**: v2.1  
**日期**: 2026-06-22  
**前置文档**: `backend/应用隔离与版本管理设计文档.md`（v1.0，存量设计基线）

本文档在 v1.0 的基础上细化工作区目录结构，采用版本级冗余设计——每个版本（Git worktree）是一个自包含的工作空间，内含独立的 `public/`（agent 文件）和 `private/`（用户私有内容），实现多人并行操作不同版本互不干扰。

---

## 1. 差异概览（相对 v1.0）

| 维度 | v1.0（存量设计） | v2.1（本文档） |
|------|-----------------|---------------|
| 目录结构 | 扁平：`{root}/{appId}` | 版本级冗余：`{root}/{app-name}/{version-dir}/` 下设 `public/`、`private/` 和项目代码 |
| public 组织 | 未定义 | 每个 agent 独立子目录，内含该 agent 的 `README.md`、`agent.md`、`skills/`、`mcp/` |
| 版本实现 | 同一目录 `git checkout` 切换 | 每个版本独立 `git worktree` 子目录，自包含 |
| 多人并行 | 不支持 | 不同用户可同时操作不同版本，互不干扰 |
| 初始化 | `git clone` + 建目录 | 每个版本创建时独立部署 public agent 和 private 目录 |
| 冗余策略 | 无 | public/ 和 private/ 冗余存储在每个版本目录下 |

### 2.2 目录职责说明

| 路径（相对于版本目录） | 职责 | 读写权限 | 生命周期 |
|--------------------------|------|---------|---------|
| `public/{agent-name}/` | 一个 agent 的完整定义目录 | 应用管理员可写，普通用户只读 | 版本创建时部署，版本删除时清理 |
| `public/{agent-name}/README.md` | 该 agent 的中文说明文档 | 同上 | 同上 |
| `public/{agent-name}/agent.md` | 该 agent 的定义文件（元信息、系统提示词等） | 同上 | 同上 |
| `public/{agent-name}/skills/` | 该 agent 专有的 skill 定义文件 | 同上 | 同上 |
| `public/{agent-name}/mcp/` | 该 agent 专有的 MCP server 配置（可选） | 同上 | 同上 |
| `private/{userId}/` | 单个用户的私有空间 | 仅该用户可读写 | 用户首次使用该版本时创建 |
| `private/{userId}/settings/` | 用户个人设置文件 | 仅该用户可读写 | 持续更新 |
| `private/{userId}/workspace/` | 用户工作区临时文件和内容 | 仅该用户可读写 | 持续更新 |
| `src/` 等 | Git 仓库项目文件 | 通过 Git 操作变更 | 版本生命周期 |

### 2.3 目录命名规则

| 目录 | 命名规则 | 示例 |
|------|---------|------|
| 应用目录 | 应用名称的 slug 化（小写、连字符与下划线保留、其他特殊字符删除） | `f-gcms`、`my_app` |
| 版本目录 | Git 分支名，其中 `/` 替换为 `_` | `main` → `main`；`feature/f-gcms_202606` → `feature_f-gcms_202606` |
| Agent 目录 | agent 名称的 slug 化 | `code-reviewer`、`test-generator` |
| 用户目录 | `private/{userId}`，userId 为 `usr_` 前缀的业务 ID | `private/usr_a1b2c3d4` |

### 2.4 冗余设计说明

每个版本目录独立存放 `public/` 和 `private/` 两份数据，原因：

1. **版本隔离**：不同版本的 agent 定义可能不同（如新版本新增 agent、旧版本移除 agent），版本级冗余确保切换版本时 agent 配置与代码一致。
2. **多人并行**：用户 A 操作 `main` 版本，用户 B 操作 `feature_f-gcms_202606` 版本，各自在独立目录工作，文件互不冲突，也不需要切换分支。
3. **简单性**：版本目录 = 完整工作空间，无需拼接多级路径或处理 inherit/override 逻辑。

代价：
- 磁盘空间增加（每个版本复制一份 agent 模板，通常 < 1MB）
- 新增版本时需要额外执行 agent 部署步骤

## 3. 核心实体与关系

### 3.1 实体定义

在 v1.0 定义的 Application / ApplicationVersion / UserApplication / Workspace 基础上，新增和明确：

| 实体 | 业务 ID 前缀 | 核心属性 | 说明 |
|------|-------------|---------|------|
| **Application** | `app_` | name, gitRepoUrl, defaultBranch, workspaceRootPath | 记录应用及其工作区根路径 |
| **ApplicationVersion** | `ver_` | appId, branchName, worktreePath, isDefault | 版本 = 分支 = worktree 目录 |
| **Workspace** | `wrk_` | appId, versionId | 每个应用-版本组合有唯一的 workspace 记录 |
| **UserApplication** | — | userId, appId, currentVersionId | 用户-应用-当前版本的对照关系 |
| **PublicAgent** | `pag_` | versionId, agentName, fileHash | 记录版本下的 agent 及文件清单 |

### 3.2 路径关系

```
总工作区根路径:      /data/workspace/                           (配置: TEST_AGENT_APPLICATION_WORKSPACE_ROOT)
应用工作区路径:      {root}/{slug(appName)}                      (如 /data/workspace/f-gcms)
版本工作区路径:      {appRootPath}/{worktreePath}                (如 /data/workspace/f-gcms/main)
agent 工作目录:      即版本工作区路径（agent 运行时的工作目录）
用户私有路径:        {versionPath}/private/{userId}/
```

### 3.3 关键约束

- 一个 Application 的 workspaceRootPath 一旦初始化不可更改
- 一个 Application 下分支名唯一 → worktreePath 唯一
- 同一版本目录下 agent 名称不重复
- ApplicationVersion 归档时，对应的 worktree 目录可选择保留或清理

| status | VARCHAR(32) | ACTIVE / ARCHIVED |

| 字段 | 类型 | 说明 |

## 6. 核心业务流程

### 6.1 应用创建

```
输入：name, description, gitRepoUrl, defaultBranch, traceId
前置：操作人具有 APP_ADMIN 或以上角色

流程：
  1. 校验 name 唯一性
  2. 规范校验：gitRepoUrl 格式、分支名字符白名单校验
  3. 创建 Application 记录（workspace_root_path 暂为空）
  4. 返回 appId

说明：应用创建不触发 workspace 初始化，初始化由独立的 initialize 接口完成。
```

### 6.2 一键初始化应用 workspace

```
输入：appId
前置：操作人具有 APP_ADMIN 或以上角色
      应用状态为 ACTIVE 且 workspace_root_path 为空（未初始化）

流程：
  1. 读取应用信息（gitRepoUrl, defaultBranch, name）
  2. 计算 workspace_root_path = {WORKSPACE_ROOT}/{slug(name)}
     如：/data/workspace/f-gcms
  3. 计算版本目录名 = branchDirName(defaultBranch)
     如：main → main，feature/v2 → feature_v2
  4. 创建目录：{workspace_root_path}/{branchDirName}/
  5. 执行 git clone --branch {defaultBranch} {gitRepoUrl} {workspace_root_path}/{branchDirName}/
     （代码直接克隆到版本目录根下）
  6. 在版本目录下创建 public/ 和 private/ 骨架（见 6.2.1）
  7. 部署公共 agent 模板到 {versionPath}/public/{agent-name}/（见 6.2.2）
  8. 创建 ApplicationVersion 记录（默认版本，status=ACTIVE, is_default=true）
  9. 创建 Workspace 记录（绑定 appId + versionId）
  10. 更新 Application.workspace_root_path
  11. 返回初始化结果

异常：
  Git clone 失败 → INTERNAL_ERROR（脱敏）
  已初始化 → CONFLICT，"该应用已初始化 workspace"
  目录创建失败 → INTERNAL_ERROR
```

#### 6.2.1 目录初始化骨架

```
{workspace_root_path}/{branchDirName}/
  public/
    .gitkeep
  private/
    .gitkeep
```

说明：版本目录本身由 `git clone` 创建（包含项目代码），额外的 `public/` 和 `private/` 在 clone 完成后手动创建。

#### 6.2.2 公共 agent 部署

初始化时，系统从【公共 agent 模板库】中复制 agent 目录到 `{versionPath}/public/`。

**模板来源**：配置项 `TEST_AGENT_PUBLIC_AGENT_TEMPLATE_DIR`，默认路径 `/opt/test-agent/agent-templates/`。

**模板目录结构**（每个 agent 一个独立子目录）：

```
/opt/test-agent/agent-templates/
├── code-reviewer/                   ← agent "code-reviewer"
│   ├── README.md                    ← 该 agent 的中文说明
│   ├── agent.md                     ← agent 定义（元信息、系统提示词）
│   ├── skills/                      ← 该 agent 专有的 skills
│   │   ├── review-changes.md
│   │   └── security-check.md
│   └── mcp/                         ← 该 agent 的 MCP 配置（可选）
│       └── config.json
├── test-generator/                  ← agent "test-generator"
│   ├── README.md
│   ├── agent.md
│   └── skills/
│       ├── unit-test.md
│       └── integration-test.md
└── bug-fixer/                       ← agent "bug-fixer"
    ├── README.md
    ├── agent.md
    └── skills/
        └── diagnose.md
```

**部署逻辑**：

```
1. 检查模板目录是否存在且非空
2. 遍历模板目录下所有子目录（每个子目录 = 一个 agent）
3. 将整个 agent 子目录复制到 {versionPath}/public/{agent-name}/
4. 对每个 agent 创建 public_agents 索引记录
5. 若模板目录不存在或为空，仅保留 public/.gitkeep
```

### 6.3 新增版本

```
输入：appId, branchName, description
前置：操作人具有 APP_ADMIN 或以上角色
      应用已初始化 workspace（workspace_root_path 不为空）
      分支名在该应用下不重复
      分支名字符白名单：[a-zA-Z0-9_/-]+

流程（Git worktree 方式）：
  1. 校验应用 workspace_root_path 不为空
  2. 计算 branchDirName = 分支名中 / → _
     如：feature/f-gcms_202606 → feature_f-gcms_202606
  3. 在应用工作区下执行：
     git worktree add {branchDirName} {branchName}
     （以 main worktree 为基础，checkout 新分支到独立目录）
     目标路径 = {workspace_root_path}/{branchDirName}/
  4. 在新版本目录下创建 public/ 和 private/ 骨架（同 6.2.1）
  5. 部署公共 agent 模板到新版本 public/ 目录（同 6.2.2）
  6. 创建 ApplicationVersion 记录（worktree_path = branchDirName）
  7. 返回 versionId

说明：每个版本是独立完整的工作空间，包含独立的 public/ 和 private/。
      不同用户可同时关联不同版本并行工作，互不干扰。

异常：
  分支名重复 → CONFLICT，"该版本已存在"
  Git worktree add 失败 → INTERNAL_ERROR
  应用未初始化 → CONFLICT，"请先初始化应用 workspace"
```

### 6.4 用户关联应用

```
输入：appId
前置：用户已登录

流程：
  1. 校验应用存在且状态为 ACTIVE
  2. 校验用户未重复关联同一应用
  3. 获取应用的默认版本（is_default=true）
  4. 创建 UserApplication 记录（current_version_id = 默认版本 ID）
  5. 返回关联结果

说明：用户关联后默认选中默认版本，后续可切换。
```

### 6.5 用户切换版本

```
输入：appId, versionId
前置：用户已关联该应用

流程：
  1. 校验版本存在且属于该应用
  2. 校验版本状态为 ACTIVE
  3. 更新 UserApplication.currentVersionId → 目标 versionId
  4. 后续 agent 运行时以新版本的 worktree_path 作为工作目录：
     工作目录 = {Application.workspace_root_path}/{ApplicationVersion.worktree_path}

说明：切换版本仅更新数据库对照关系。由于版本以独立 worktree 存在，
      无需执行 git checkout，切换即时生效。
      用户 A 切换不影响用户 B。

异常：
  版本不存在 → NOT_FOUND
  用户未关联该应用 → FORBIDDEN
  版本已归档 → CONFLICT
```

### 6.6 多人并行场景示例

```
用户 A 关联 f-gcms，当前版本 main
  → agent 工作目录：/data/workspace/f-gcms/main/
  → 该目录下包含 main 分支代码 + main 版本的 public agent

用户 B 关联 f-gcms，当前版本 feature_f-gcms_202606
  → agent 工作目录：/data/workspace/f-gcms/feature_f-gcms_202606/
  → 该目录下包含 feature 分支代码 + 该版本的 public agent

两个用户完全独立，可同时发起 agent Run，互不干扰。
用户的 private/ 数据（sessions、settings）也存储在各自版本目录下。
```

### 6.7 公共 agent 文件管理（按版本管理）

| 操作 | 权限 | 说明 |
|------|------|------|
| 列出某版本下所有 agent | 所有关联用户 | 读取 `public_agents` 表，按 `version_id` 过滤 |
| 查看某 agent 详情（含 README、skills 列表） | 所有关联用户 | 读取版本目录 `public/{agent-name}/` 下文件 |

| 删除 agent | APP_ADMIN+ | 删除版本目录 + 标记 public_agents 记录为 ARCHIVED |

## 8. 安全设计

| 关注点 | 措施 |
|--------|------|
| 目录越权 | 所有文件操作必须校验目标路径在 `{version_path}` 之内；version_path 由数据库 trust 而不由用户传参 |
| public 目录写保护 | 非 APP_ADMIN 用户对 `public/` 不可写 |
| private 目录隔离 | 每个用户只能访问 `private/{selfUserId}/` 下的内容，不可跨用户 |
| private 大小限制 | 单用户单版本 private 目录上限 500MB，配置化 |
| Git 认证凭据 | 通过环境变量 `GIT_USERNAME` / `GIT_TOKEN` 注入，不存储到数据库 |
| 分支名校验 | 白名单 `[a-zA-Z0-9_/-]+`，长度上限 255，拒绝命令注入 |
| 日志脱敏 | Git 命令输出不全量记录到日志，错误信息脱敏后返回前端 |
| 审计追踪 | 初始化、版本创建、agent 增删改、版本切换均记录 traceId + 操作人 + 结果 |
| 越权防护 | Controller 入口校验用户身份和角色，不依赖前端判断 |
